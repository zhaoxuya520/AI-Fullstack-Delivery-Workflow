# RULES

## 1. 通用执行规则

1. 不直接跳过需求澄清进入编码。
2. 每次交付必须说明输入、输出、验证方式。
3. 路由不明确时，先澄清任务目标和期望产物。
4. 涉及多个岗位时，先拆分上下游工作流，不把所有任务塞进一个工作流。
5. 优先复用已有模板、工具索引和 field-journal 经验。
6. 任务完成后，必须按需生成技术文档或报告。
7. 有可复用经验时，回写到 `field-journal/`。
8. 不把临时结论当成长期规则；长期规则必须写入 `RULES.md` 或对应工作流文档。

## 2. 编码与工程规则

1. 修改代码前先识别项目技术栈和现有结构。
2. 优先修改现有文件，不随意新增重复抽象。
3. 代码类交付必须给出运行、测试或验证方式。
4. 不硬编码密钥、Token、密码、私有地址等敏感信息。
5. 不引入无必要的新依赖。
6. 涉及外部输入时必须考虑注入、越权、XSS、SSRF、路径穿越等风险。

## 3. 数据库规则

1. 涉及数据库迁移时，必须说明备份、回滚和兼容性。
2. 生产数据变更必须优先考虑可回滚和可审计。
3. 新增索引必须说明查询场景和写入影响。
4. 删除字段、表、索引属于高风险操作，必须单独确认。

## 4. 部署与运维规则

1. 涉及部署时，必须包含健康检查和回滚方案。
2. 涉及生产问题时，必须先止血，再定位根因。
3. CI/CD 变更必须说明触发条件、影响环境和失败处理。
4. 环境变量、密钥和证书不得写入公开文档或代码。

## 5. 测试规则

1. 功能交付至少覆盖核心成功路径。
2. 涉及权限、支付、数据删除、状态流转时必须覆盖异常路径。
3. Bug 修复必须说明复现方式和回归方式。
4. 自动化测试不应依赖不可控外部状态，除非任务明确要求端到端真实环境。

## 6. 安全任务前置要求

所有安全、逆向、渗透、漏洞验证任务必须先确认：

- 目标是否属于授权范围
- 是否有书面或明确授权
- 是否允许主动扫描
- 是否允许漏洞利用验证
- 是否允许生成 PoC
- 是否涉及第三方系统
- 是否允许保存或展示敏感证据

未经授权，不执行破坏性、拒绝服务、批量攻击、供应链投毒、恶意规避检测等操作。

## 7. 自进化规则

1. 每个任务完成后必须按根 `EVOLUTION.md` 和对应工作流 `EVOLUTION.md` 执行自进化检查。
2. 有可复用经验时，优先写入对应工作流 `field-journal/`。
3. 新增任务类型或关键词时，更新对应 `routing.md`；影响全局时同步根 `routing.md`。
4. 使用新工具或命令时，更新对应 `tool-index.md`；影响全局时同步根 `tool-index.md`。
5. 发现新常见坑时，更新对应 `pitfalls.md`。
6. 产出可复用格式时，放入对应 `templates/`。
7. 发现长期有价值的方法论、规范或速查表时，放入对应 `references/`。
8. 可重复步骤应该沉淀为 `scripts/`，而不是只写在聊天记录里。

## 8. 文档规则

1. 文档必须明确背景、输入、步骤、输出、风险和限制。
2. 安全类报告必须包含授权范围、复现步骤、风险等级、证据和修复建议。
3. 部署类文档必须包含环境要求、命令、验证方式和回滚方案。
4. 复盘类文档必须包含时间线、根因、影响、修复和后续改进。
