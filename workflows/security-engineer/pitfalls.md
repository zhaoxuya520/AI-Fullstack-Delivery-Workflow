# 安全工程师工作流常见坑

## 1. 只跑工具不手动

- 表现：只跑 ZAP / Snyk，不做手动测试。
- 风险：漏业务逻辑漏洞（IDOR / 越权）。
- 避免：工具 + 手动结合。

## 2. 只报漏洞不给修复

- 表现：报告只写"存在 XSS"。
- 风险：开发不知道怎么修。
- 避免：每个漏洞必须附修复建议 + 验证方式。

## 3. 安全最后做

- 表现：上线前才做安全审查。
- 风险：返工成本高。
- 避免：设计阶段做威胁建模，PR 阶段做代码审查。

## 4. 忽略 IDOR

- 表现：不测跨用户访问。
- 风险：最常见的高危漏洞。
- 避免：每个端点都测越权。

## 5. 不测越权

- 表现：只测自己角色。
- 风险：A 能改 B 数据。
- 避免：权限矩阵全覆盖。

## 6. 依赖不扫描

- 表现：不跑 npm audit / Snyk。
- 风险：已知漏洞进生产。
- 避免：CI 集成依赖扫描。

## 7. 报告太技术

- 表现：PM 看不懂。
- 风险：决策延误。
- 避免：摘要用业务语言，详情用技术语言。

## 8. 不验证修复

- 表现：开发说修了就信了。
- 风险：没真修好。
- 避免：每个修复必须重新验证。

## 9. 不做威胁建模

- 表现：不知道重点在哪。
- 风险：测试方向错。
- 避免：新功能必做 STRIDE。

## 10. 安全报告泄露

- 表现：报告发到公开频道。
- 风险：攻击者利用。
- 避免：报告分发范围受控。
