# 交互状态指南

## 1. 为什么要设计状态

真实界面不是静态截图。页面会加载失败、数据为空、用户无权限、提交出错、网络中断。状态设计用于保证用户在任何情况下都知道发生了什么、能做什么、如何恢复。

---

## 2. 页面级状态

| 状态 | 设计目标 | 必须说明 |
|---|---|---|
| 加载中 | 告诉用户系统正在处理 | 加载区域、是否可取消、骨架屏/Spinner |
| 空数据 | 告诉用户没有内容及下一步 | 空原因、引导动作、创建入口 |
| 错误 | 告诉用户失败原因 | 错误原因、重试方式、是否保留输入 |
| 权限不足 | 告诉用户不能操作的原因 | 缺少什么权限、是否能申请 |
| 成功 | 确认操作已完成 | 成功信息、下一步入口 |

---

## 3. 表单状态

```text
默认：字段说明清楚。
聚焦：焦点明显。
输入中：必要时实时校验。
错误：指出字段、原因和修正方式。
提交中：防止重复提交。
提交成功：明确结果。
提交失败：保留输入并给出恢复路径。
```

---

## 4. 危险操作状态

危险操作包括删除、禁用、支付、审批、覆盖、批量修改。

```text
是否需要二次确认？
是否需要输入确认文本？
是否可以撤销？
是否需要说明影响范围？
是否需要审计或日志？
```

---

## 5. 权限状态

```text
隐藏：用户不应知道此能力存在。
禁用：用户知道能力存在，但当前不可用。
提示：用户点击后说明原因和申请路径。
只读：用户可查看但不可编辑。
```

---

## 6. 状态交接给前端

每个状态至少写清：

```text
触发条件
展示内容
可用操作
API / 错误码来源
恢复路径
验收标准
```
