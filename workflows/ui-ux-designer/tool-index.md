## 3. 方法工具

| 方法 | 用途 | 适用场景 |
|---|---|---|
| 用户任务分析 | 明确用户要完成什么 | 所有页面设计 |
| 用户流程 | 串联入口、操作、反馈和结果 | 多步骤任务 |
| 信息架构 | 组织页面、导航和内容层级 | 多页面、多模块系统 |
| 线框图 | 快速表达布局和信息优先级 | 早期设计、交接前 |
| 组件化设计 | 拆分可复用 UI 单元 | 前端交接、设计系统 |
| 状态设计 | 覆盖加载、空、错误、权限、成功状态 | 所有交互页面 |
| 响应式设计 | 适配桌面、平板、移动端 | Web、小程序、多端系统 |
| 可访问性检查 | 提升键盘、焦点、对比度、语义可用性 | 面向广泛用户的产品 |
| 行业对标 | 从大厂/热门项目提炼成熟模式 | 企业后台、SaaS、复杂交互 |
| 模式库沉淀 | 把重复页面/组件整理成可复用模式 | 多页面产品、长期项目 |
| 可用性评审 | 发现理解、操作、反馈和恢复问题 | 原型完成后、开发前 |
| 视觉风格定义 | 明确页面气质、色彩、排版、组件形态和动效边界 | 页面设计前、品牌风格不清时 |
| 反 AI 化检查 | 避免无意义渐变、假 3D、漂浮发光球和模板化文案 | 设计评审、前端交接前 |
| GitHub 项目拆解 | 从热门开源 UI 项目提炼组件、动效、主题和品类风格 | 需要真实可落地参考时 |

## 7. 使用原则

```text
1. 先明确用户任务，再定义页面结构。
2. 先参考成熟模式，再按本项目约束适配，不能照抄视觉。
3. 企业后台优先检查表单、表格、导航、空状态、权限状态。
4. 页面设计前先定义视觉风格和明确不用的 AI 模板化效果。
5. 使用 Radix/shadcn 等生态时，必须说明键盘、焦点和 ARIA 行为。
6. 每个页面必须有目标、入口、主操作和反馈。
7. 每个核心组件必须有状态说明。
8. 页面字段要能对接 API 和数据库。
9. 前端交接必须包含组件、状态、响应式、可访问性和验收标准。
10. 可复用页面/组件模式要沉淀到 `ui-pattern-library-template.md`。
```
