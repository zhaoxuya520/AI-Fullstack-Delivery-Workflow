# UI/UX 设计工作流路由

## 定位

本文件定义哪些任务应进入 UI/UX 设计工作流，以及进入后应该产出什么。

UI/UX 设计工作流负责把 PRD、用户故事和业务流程转化为页面结构、用户流程、交互状态、组件说明和前端可实现的设计交接包。

## 按任务类型路由

| 用户任务 | 本工作流动作 | 输出 |
|---|---|---|
| “帮我设计页面” | 梳理页面目标、结构和组件 | 页面设计说明 |
| “这个流程怎么走” | 设计用户路径和关键反馈 | 用户流程图 |
| “帮我画线框图” | 输出文本线框和布局说明 | 线框图文本稿 |
| “前端需要哪些组件” | 拆分组件、状态和交互 | 组件清单、状态说明 |
| “这个页面有哪些状态” | 补齐加载、空、错误、权限等状态 | 交互状态说明 |
| “移动端怎么适配” | 定义响应式规则 | 响应式说明 |
| “帮我交给前端” | 汇总页面、组件、状态、验收 | 前端交接包 |
| “这个设计好不好用” | 做可用性检查 | 可用性评审报告 |

## 进入前补问规则

如果缺少目标用户，先问：

```text
这个页面主要给谁使用？普通用户、管理员、商家、内部人员，还是其他角色？
```

如果缺少页面目标，先问：

```text
用户进入这个页面后，最重要的任务是什么？
```

如果缺少目标终端，先问：

```text
这个页面主要运行在桌面端、移动端、小程序、平板还是响应式 Web？
```

如果缺少数据字段，先问：

```text
页面必须展示哪些数据？这些字段是否已有 API 或数据库来源？
```

## 转出规则

| 条件 | 转出到 |
|---|---|
| 需求、用户目标、业务规则不清 | 产品经理工作流 |
| 需要排期和里程碑 | 项目经理工作流 |
| 页面需要的数据字段不清 | API 设计工作流 |
| 业务实体和数据来源不清 | 数据库工程师工作流 |
| 需要实现页面代码 | 前端工程师工作流 |
| 需要测试用例 | QA 工作流 |
| 涉及敏感信息展示、危险操作、权限边界 | 安全工程师工作流 |
| 需要用户文档 | 技术文档工作流 |

---

## 路由未命中

如果任务主要是业务需求、排期、API、数据库、代码实现、测试、安全或部署，不应强行进入 UI/UX 设计工作流，应返回根 `routing.md` 重新选择。
