# 项目管理实战指南

> 面向 APP / 小程序 / 网页项目的迭代管理、风险控制、团队协作。

## 1. 迭代规划

### Sprint 节奏

```text
2 周迭代（推荐）：
  Day 1: Sprint Planning（2h）
  Day 1-9: 开发 + 每日站会（15min）
  Day 9: Code Freeze + 提测
  Day 10: Sprint Review + 演示
  Day 10: Sprint Retro（1h）

1 周迭代（快速验证期）：
  Day 1: Planning（1h）
  Day 1-4: 开发
  Day 4: 提测
  Day 5: Review + Retro

每日站会（15min 严格）：
  - 昨天做了什么
  - 今天要做什么
  - 有什么阻塞
```

### 任务拆解原则

```text
SMART 任务：
  S - 具体（不要"优化性能"，要"首屏加载从3s降到1.5s"）
  M - 可衡量（有验收标准）
  A - 可分配（一个人能做完）
  R - 有关联（指向里程碑）
  T - 有时限（≤ 3天，超过就继续拆）

任务粒度：
  理想: 0.5~2 天/任务
  最大: 3 天（超过必须拆分）
  最小: 2 小时（太小合并）

依赖标注：
  每个任务标注前置依赖
  识别关键路径（最长依赖链）
  并行安排无依赖任务
```

## 2. 风险管理

### 风险矩阵

```text
           │ 低概率    │ 中概率    │ 高概率
───────────┼───────────┼───────────┼──────────
高影响     │ 关注+预案 │ 高优处理  │ 立即处理
中影响     │ 定期检查  │ 关注+预案 │ 高优处理
低影响     │ 接受      │ 定期检查  │ 关注
```

### 常见项目风险

| 风险 | 应对 |
|------|------|
| 需求变更频繁 | 变更流程 + 影响评估 + 缓冲时间 |
| 关键人员请假/离职 | 知识共享 + 文档 + 结对 |
| 技术方案不确定 | Spike（技术调研时间盒） |
| 第三方依赖延期 | 提前 Mock + 备选方案 |
| 联调阻塞 | API 契约先行 + Mock Server |
| 测试发现大量 Bug | 预留修复缓冲 + 分级处理 |

## 3. 里程碑模板

```text
项目: [项目名]
总工期: [开始日期] ~ [上线日期]

M1 - 需求冻结 [日期]
  □ PRD 评审通过
  □ 设计稿交付
  □ API 契约确认
  □ 技术方案确认

M2 - 开发完成 [日期]
  □ 前端页面完成
  □ 后端接口完成
  □ 联调通过
  □ 单元测试覆盖

M3 - 测试完成 [日期]
  □ 功能测试通过
  □ 兼容性测试通过
  □ 性能测试达标
  □ 安全扫描通过
  □ P0/P1 Bug 清零

M4 - 上线 [日期]
  □ 上线 checklist 通过
  □ 灰度发布 10% → 50% → 100%
  □ 监控告警配置
  □ 回滚方案确认
```

## 4. 协作规范

```text
沟通原则：
  - 异步优先（飞书/Slack 文字 > 会议）
  - 会议必须有议程和结论
  - 阻塞问题 4h 内升级

文档规范：
  - PRD 用 Notion/飞书文档
  - 技术方案用 Markdown（Git 管理）
  - 会议纪要当天归档

状态同步：
  - 每日站会（团队）
  - 周报（向上）
  - 每两周 Sprint Review（向干系人）
```
