# 项目管理方法参考

## 1. WBS 任务分解

```text
项目目标
→ 阶段
→ 工作流
→ 任务
→ 验收标准
```

原则：

```text
每个任务有唯一主要负责工作流。
每个任务有明确输入、输出和验收标准。
超过 60 分钟的任务继续拆分。
人工等待节点单独标注，不混入 AI 执行时间。
```

---

## 2. 依赖类型

```text
FS：完成-开始，A 完成后 B 开始。
SS：开始-开始，A 开始后 B 可开始。
FF：完成-完成，A 完成后 B 才能完成。
外部依赖：审批、第三方、环境、账号、证书等不可控节点。
```

---

## 3. 关键路径

关键路径是决定项目最短完成时间的最长依赖链。

```text
识别所有任务
→ 标注依赖
→ 计算最长链路
→ 标记关键路径任务
→ 优先监控关键路径风险
```

关键路径任务不能随意延迟；关键路径以外的任务可以利用缓冲和并行。

---

## 4. 里程碑

好的里程碑必须可验收：

```text
有明确产物
有明确时间点
有明确负责人或工作流
有门禁检查
通过/不通过标准清楚
```

---

## 5. 任务状态机

```text
todo → in_progress → review → done
  ↓        ↓             ↓
skipped  blocked       rework → in_progress
          ↓
       waiting_human → in_progress
```

状态规则：

```text
blocked 必须写原因和下一步。
waiting_human 必须写等待谁、等待什么。
review 不通过进入 rework。
skipped 必须写原因，且不能跳过门禁任务。
```

---

## 6. Cycle 管理

AI 工作流适合短周期交付：

```text
Micro Cycle：30~60 分钟，单个功能点。
Standard Cycle：2~4 小时，完整模块。
Extended Cycle：1 天，跨模块功能。
```

Cycle 规则：

```text
开始前明确目标和验收标准。
Cycle 内不随意加需求。
时间到但未完成，优先砍范围而不是降低质量。
Cycle 结束做快速复盘。
```

---

## 7. 项目复盘

复盘关注：

```text
目标是否达成？
计划和实际差异在哪里？
哪些依赖估错？
哪些交接导致返工？
哪些门禁没有拦住问题？
哪些经验应写入 field-journal？
```
