# 项目经理工作流路由

## 定位

本文件定义哪些任务应进入项目经理工作流，以及进入后应该产出什么。

项目经理工作流负责把 PRD、功能范围和交付目标转化为可执行计划，重点处理任务拆解、依赖、排期、门禁、风险、变更和跨工作流协同。

## 按任务类型路由

| 用户任务 | 本工作流动作 | 输出 |
|---|---|---|
| “帮我排期” | 识别范围、依赖、资源和关键路径 | 里程碑计划、任务列表 |
| “帮我拆任务” | 按 WBS 拆成工作流任务 | WBS、负责人、验收标准 |
| “这个项目怎么推进” | 选择编排模式并规划阶段 | 项目执行计划 |
| “哪些可以并行” | 分析依赖和并发安全 | DAG、并行组、关键路径 |
| “上线前要检查什么” | 定义门禁和检查项 | 发布/上线 checklist |
| “现在进度怎么样” | 汇总状态、阻塞和风险 | 进度报告 |
| “这个变更影响什么” | 做变更影响评估 | 影响范围、计划调整建议 |
| “项目有什么风险” | 建立风险登记册 | 风险清单、缓解方案 |
| “项目结束帮我复盘” | 汇总目标、时间线、问题和经验 | 项目复盘报告 |
| “多个工作流怎么协作” | 定义交接点和交付物 | 工作流交接计划 |

## 进入前补问规则

如果缺少需求范围，先问：

```text
这个项目的 PRD、功能范围或用户故事在哪里？如果还没有，应先进入产品经理工作流。
```

如果缺少 deadline，先问：

```text
这个项目有硬性上线时间吗？如果没有，是希望尽快交付还是优先保证质量？
```

如果缺少资源信息，先问：

```text
有哪些工作流或人员可以参与？是否有只能人工处理的节点？
```

如果缺少验收标准，先问：

```text
项目完成的判断标准是什么？谁验收？必须通过哪些门禁？
```

## 转出规则

| 条件 | 转出到 |
|---|---|
| 需求范围不清 | 产品经理工作流 |
| 页面设计未完成 | UI/UX 设计工作流 |
| API 契约缺失 | API 设计工作流 |
| 数据模型缺失 | 数据库工程师工作流 |
| 需要具体代码实现 | 前端 / 后端 / 全栈工作流 |
| 需要测试设计或执行 | QA / 自动化测试工作流 |
| 涉及权限、敏感数据、漏洞风险 | 安全工程师工作流 |
| 需要环境、部署、CI/CD | DevOps 工作流 |
| 需要上线监控、告警、SLO | SRE/运维工作流 |
| 需要正式交付文档 | 技术文档工作流 |

---

## 路由未命中

如果任务主要是需求定义、设计、代码实现、测试、安全验证、部署或数据分析，不应强行进入项目经理工作流，应返回根 `routing.md` 重新选择。
