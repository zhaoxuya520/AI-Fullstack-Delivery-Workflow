# 项目经理工作流常见坑

## 1. 没有 PRD 就开始排期

### 表现

需求范围不清、验收标准不清，却直接拆任务和估工期。

### 风险

- 任务反复变化
- 排期没有依据
- 下游工作流返工

### 避免方式

先检查：

```text
PRD 或功能范围是否存在？
验收标准是否明确？
非目标范围是否明确？
```

---

## 2. 只列任务，不标依赖

### 表现

任务列表看起来完整，但不知道谁先谁后。

### 风险

- 前置任务未完成，下游提前开工
- 关键路径不可见
- 并行和串行判断错误

### 避免方式

每个任务必须写：

```text
负责工作流
前置依赖
输出产物
验收标准
是否在关键路径上
```

---

## 3. 过度并行

### 表现

为了快，把 API、数据库、后端、前端同时开工。

### 风险

- 契约未定导致字段不一致
- 数据结构变化引发返工
- 联调成本上升

### 避免方式

并行前检查共享契约：

```text
API 契约是否稳定？
数据库 schema 是否稳定？
业务规则是否稳定？
错误码和权限是否定义？
```

---

## 4. 没有联调和回归缓冲

### 表现

排期只计算开发时间，不给联调、Bug 修复、回归测试留时间。

### 风险

- 最后阶段集中爆雷
- 上线延期
- 质量门禁被迫降低

### 避免方式

每个跨工作流项目至少预留：

```text
联调缓冲
回归缓冲
部署验证缓冲
返工缓冲
```

---

## 5. 安全和部署太晚介入

### 表现

开发完成后才让安全和 DevOps 看。

### 风险

- 权限、敏感数据、部署架构问题难以修正
- 上线前发现高风险问题

### 避免方式

在计划中提前设置：

```text
需求阶段：安全约束识别
设计阶段：权限和数据风险检查
开发阶段：安全检查并行
上线阶段：部署和回滚门禁
```

---

## 6. 里程碑不可验收

### 表现

里程碑写成“开发完成 50%”“基本可用”。

### 风险

- 无法判断是否通过
- 项目状态虚假乐观

### 避免方式

里程碑必须包含：

```text
具体产物
验收标准
门禁检查
负责人
时间点
```

---

## 7. 风险只记录不跟踪

### 表现

风险清单写完后不更新状态。

### 风险

风险变成文档装饰，真正发生时没人处理。

### 避免方式

风险登记册必须包含：

```text
概率
影响
触发条件
缓解方案
负责人
状态
下次检查时间
```

---

## 8. 变更不评估影响

### 表现

中途加需求、改规则、换方案，但计划不变。

### 风险

- 范围膨胀
- 工期失真
- 下游工作流漏改

### 避免方式

变更必须检查：

```text
范围影响
任务影响
依赖影响
工期影响
风险影响
门禁影响
文档影响
```

---

## 9. 交接只传文件不传上下文

### 表现

上游只丢一个文档给下游，不说明决策背景和注意事项。

### 风险

- 下游误解需求
- 已知限制被遗漏
- 返工责任不清

### 避免方式

每次交接必须包含：

```text
Artifact：具体产物
Context：背景、决策、限制
Acceptance：下游验收标准
```

---

## 10. 阻塞后沉默等待

### 表现

任务 blocked 后没有原因、负责人、下一步和超时策略。

### 风险

- 项目停滞但没人知道
- 关键路径延迟扩大

### 避免方式

阻塞必须写清：

```text
阻塞原因
影响范围
负责人
下一步动作
预计解除时间
是否需要用户决策
```

---

## 11. 状态过于粗糙

### 表现

只有“未开始 / 进行中 / 完成”。

### 风险

review、rework、blocked、waiting_human 被隐藏。

### 避免方式

使用状态机：

```text
todo / in_progress / blocked / waiting_human / review / rework / done / skipped
```

---

## 12. 项目结束不复盘

### 表现

上线后直接进入下一个项目。

### 风险

估算偏差、交接问题、风险遗漏无法沉淀。

### 避免方式

项目结束后必须复盘：

```text
目标是否达成？
计划和实际差异？
哪些阻塞重复出现？
哪些模板需要更新？
哪些经验要写入 field-journal？
```
