# 工作流编排指南

## 1. 顺序编排

```text
A → B → C → D
```

适合：强依赖链路。

示例：

```text
PRD → API 契约 → 后端开发 → 联调 → QA → 部署
```

---

## 2. 并发编排

```text
A → [B, C, D] → 汇总
```

适合：多个任务依赖同一个稳定输入，但彼此不互相依赖。

可并发示例：

```text
API 契约稳定后：前端和后端并行。
PRD 稳定后：QA 用例设计和开发可部分并行。
后端完成后：QA 和安全检查可部分并行。
```

---

## 3. 交接编排

当前工作流完成后，根据结果决定下一个工作流。

```text
Bug 分析 → 判断类型 → 前端 / 后端 / 数据库 / DevOps
```

适合：问题类型不确定、需要动态路由的任务。

---

## 4. 动态编排

项目经理根据中间结果重新规划任务。

适合：

```text
线上故障
探索性原型
不确定技术方案
需求频繁变化
多工作流协同问题
```

---

## 5. 混合编排

实际项目通常是混合模式：

```text
产品经理
  ↓
API 设计 + 数据库设计 + UI/UX 设计
  ↓
前端 + 后端
  ↓
联调
  ↓
QA + 安全
  ↓
DevOps + 技术文档
```

---

## 6. 并发安全规则

并发前必须确认：

```text
不会同时修改同一文件。
共享契约已稳定。
有合并和冲突处理规则。
下游能独立验证自己的产物。
```

禁止并发：

```text
API 未定就做后端。
数据模型未定就做持久化逻辑。
权限规则未定就做安全敏感功能。
测试环境未准备好就安排上线验证。
```
