# 交付门禁指南

## 1. 门禁目的

门禁不是流程装饰，而是防止低质量输入进入下一阶段。

```text
需求不清 → 不进入设计/开发
契约不清 → 不进入前后端并发
测试不过 → 不进入上线
安全风险未处理 → 不进入发布
```

---

## 2. 需求门禁

必须确认：

```text
PRD 已确认
目标用户和核心问题明确
MVP 和非目标范围明确
验收标准可测试
权限、异常、边界场景已说明
```

---

## 3. 设计门禁

必须确认：

```text
UI/UX 关键路径和状态明确
API 契约明确
数据库实体和生命周期明确
权限模型明确
错误码和异常处理明确
```

---

## 4. 开发门禁

必须确认：

```text
核心功能完成
关键测试通过
接口联调通过
配置和环境说明清楚
没有明显安全风险
```

---

## 5. 测试门禁

必须确认：

```text
成功路径通过
失败路径通过
边界条件通过
高优先级 Bug 已修复
已知问题记录并被接受
```

---

## 6. 上线门禁

必须确认：

```text
部署步骤明确
回滚方案明确
监控和日志就绪
健康检查通过
安全检查通过或风险被明确接受
相关方已通知
```

---

## 7. 门禁结果

```text
通过：进入下一阶段。
带问题通过：记录风险、负责人和补救时间。
不通过：退回上游 rework。
跳过：必须由用户明确确认，并记录原因。
```
