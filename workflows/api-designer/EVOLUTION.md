# API 设计工作流自进化协议

## 目标

API 设计工作流在完成真实接口设计任务后，应判断是否产生可复用的契约模式、错误码规则、权限模型、Mock 经验或交接规范，并把经验沉淀回本工作流。

---

## 需要更新的文件

当任务产生可复用知识时，检查并更新：

```text
field-journal/      # 真实任务经验
pitfalls.md         # 常见失败和预防方式
routing.md          # 新关键词、触发条件、输入输出
tool-index.md       # 工具、脚本、模板、参考资料
references/         # API 方法、规范、风格指南
scripts/            # 可重复自动化检查
WORKFLOW.md         # 流程、输入、输出、质量门禁
```

---

## API 专项沉淀触发

任务完成后检查：

```text
是否形成新的资源建模规则？
是否出现新的端点命名模式？
是否发现新的错误码或状态码坑？
是否出现新的认证、鉴权、租户或字段权限规则？
是否需要统一分页、筛选、排序或搜索模式？
是否出现新的版本兼容或弃用策略？
是否涉及幂等、重试、限流或并发冲突经验？
是否涉及 Webhook、异步任务、签名或重放防护经验？
是否形成新的 Mock 或联调交接模式？
是否需要更新检查脚本？
```

---

## 完成检查

```text
1. 新的可复用经验？→ 写入 field-journal。
2. 新的常见坑？→ 更新 pitfalls.md。
3. 新的路由关键词或任务类型？→ 更新 routing.md，必要时更新根 routing.md。
4. 新的工具或脚本？→ 更新 tool-index.md 和 scripts/README.md。
5. 新的交付格式？→ 新增或更新 templates/。
6. 新的长期方法资料？→ 新增或更新 references/。
7. 标准流程变化？→ 更新 WORKFLOW.md。
8. 影响其他工作流？→ 更新相关工作流交接说明。
```

---

## 不记录内容

```text
临时任务状态
未经验证的猜测
一次性接口细节
密钥、Token、账号、客户数据
敏感内部地址
可从代码或 git 历史直接恢复的信息
```

---

## Field Journal 流程

```text
复制 field-journal/_template.md
→ 创建日期命名条目
→ 记录任务背景
→ 记录问题和解决方案
→ 记录验证方式
→ 提炼可复用经验
→ 更新 field-journal/_index.md
```
