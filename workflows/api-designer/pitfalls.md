# API 设计工作流常见坑

## 1. 业务对象不清就开始列接口

### 表现

直接写 `/createOrder`、`/getList`、`/submit`，没有先定义资源、状态和关系。

### 风险

- API 路径混乱
- 后续版本难演进
- 前端和后端对业务对象理解不一致

### 避免方式

先完成资源模型：

```text
资源名称
资源关系
资源状态
生命周期
可执行操作
动作化端点例外
```

---

## 2. 把数据库表结构暴露成 API

### 表现

响应字段直接等于数据库字段，包含内部 ID、软删除字段、内部状态码或审计字段。

### 风险

- 内部实现被外部绑定
- 数据库改动变成 API 破坏性变更
- 敏感字段泄露

### 避免方式

API 字段必须按调用方任务设计，并标注：

```text
字段含义
来源
是否敏感
是否稳定
是否可为空
前端展示方式
```

---

## 3. 只设计成功响应

### 表现

只写 200 返回示例，不写空结果、校验失败、权限不足、资源不存在、冲突、限流和服务异常。

### 风险

- 前端错误处理只能猜
- QA 漏测失败路径
- 后端实现不一致

### 避免方式

每个核心端点至少覆盖：

```text
成功
空结果
参数错误
认证失败
权限不足
资源不存在
资源冲突
限流
服务异常
```

---

## 4. 错误码只有文案

### 表现

只返回“操作失败”“参数错误”，没有业务错误码、触发条件和调用方处理建议。

### 风险

- 前端无法精确提示
- QA 无法断言错误类型
- 调用方无法自动恢复

### 避免方式

错误码表必须包含：

```text
HTTP 状态码
业务错误码
错误文案
触发条件
调用方处理建议
是否可重试
trace/request id
```

---

## 5. 用 200 包装所有错误

### 表现

业务失败、权限失败、资源不存在都返回 HTTP 200，再在 body 中塞 success=false。

### 风险

- HTTP 语义丢失
- 监控和网关难以识别失败
- SDK 和调用方处理复杂

### 避免方式

按语义使用状态码：

```text
400 参数错误
401 未认证
403 无权限
404 资源不存在
409 冲突
422 业务校验失败
429 限流
500 服务异常
```

---

## 6. 权限只写“需登录”

### 表现

所有接口权限都写需登录，不区分角色、资源归属、租户和字段级权限。

### 风险

- 越权风险
- 前端展示错误入口
- 安全评审无法判断边界

### 避免方式

权限矩阵必须定义：

```text
角色
资源范围
租户/组织
可读/可创建/可更新/可删除
字段级限制
无权限响应
```

---

## 7. 分页、筛选、排序风格不统一

### 表现

有的接口用 page/pageSize，有的用 offset/limit，有的排序写 `sort=created_at`，有的写 `orderBy`。

### 风险

- 前端封装困难
- 文档和测试重复成本高
- 后端查询能力不一致

### 避免方式

统一定义：

```text
分页模式
默认页大小和最大页大小
允许筛选字段
允许排序字段
默认排序
搜索范围
空结果响应
```

---

## 8. POST / PUT / PATCH 语义混乱

### 表现

所有修改都用 POST，或者 PUT/PATCH 不区分全量和局部更新。

### 风险

- 调用方难以理解副作用
- 幂等和重试策略不清
- 缓存、代理和日志语义混乱

### 避免方式

```text
POST：创建或非幂等动作
PUT：整体替换或幂等写入
PATCH：局部更新
DELETE：删除或撤销资源
```

---

## 9. 缺少幂等和重试策略

### 表现

创建订单、提交审批、支付、扣减库存等接口没有 Idempotency-Key 或业务去重规则。

### 风险

- 网络重试造成重复提交
- 扣款、扣减、审批重复执行
- 调用方不敢重试

### 避免方式

高风险写操作必须定义：

```text
幂等键
去重窗口
重复请求响应
超时后查询方式
可重试错误
不可重试错误
并发冲突处理
```

---

## 10. Webhook 没有安全和可靠性设计

### 表现

只写回调 URL 和 payload，没有签名、事件 ID、重试、重放防护和失败处理。

### 风险

- 伪造事件
- 重放攻击
- 接收方漏处理或重复处理
- 事件顺序不可控

### 避免方式

Webhook 必须定义：

```text
事件 ID
事件类型
签名算法
时间戳
重放窗口
重试策略
幂等处理
失败告警
死信或补偿
```

---

## 11. Mock 与契约不一致

### 表现

Mock 数据手写后没有同步 Schema，字段名、类型、错误结构和真实后端不一致。

### 风险

- 前端基于错误数据开发
- 联调返工
- QA 测试用例失真

### 避免方式

```text
Mock 示例必须来自 OpenAPI Schema。
Mock 必须覆盖成功、空、错误、权限、冲突和限流。
Mock 修改必须同步契约。
```

---

## 12. OpenAPI 和 Markdown 说明不一致

### 表现

Markdown 写了字段 A，OpenAPI 写字段 B；错误码、示例、权限说明不同步。

### 风险

- 前后端各看一份文档，各实现一套
- 自动生成文档失真
- 联调责任不清

### 避免方式

```text
OpenAPI 作为机器可读契约。
Markdown 只解释设计意图、边界和交接。
同一字段、错误、状态必须保持一致。
```

---

## 13. 忽略版本和兼容性

### 表现

随意删除字段、改枚举含义、修改必填规则，没有弃用和迁移说明。

### 风险

- 老客户端被破坏
- 第三方调用方不可控
- 发布回滚困难

### 避免方式

所有变更必须标注：

```text
兼容 / 不兼容
影响端点
影响调用方
迁移方式
弃用时间
回滚方式
```

---

## 14. 交接只给端点，不给测试和恢复路径

### 表现

只发一份接口列表，没有错误路径、权限矩阵、Mock、联调步骤和验收标准。

### 风险

- 前端无法独立开发
- QA 无法覆盖边界
- 安全无法评估越权

### 避免方式

交接包必须包含：

```text
端点清单
请求响应示例
错误码表
权限矩阵
Mock 地址或数据
联调步骤
验收标准
待确认问题
```
