# Postmortem 模板

## 事故信息
```text
事故 ID：INC-YYYY-MM-DD-XXX
日期：
严重度：P1 / P2
持续时间：
影响：[用户数 / 功能]
作者：
评审人：
```

## 摘要（3 句话）
[什么坏了 + 影响多大 + 怎么修的]

## 时间线
| 时间 (UTC) | 事件 |
|---|---|
| 14:30 | 告警触发：错误率 > 5% |
| 14:32 | On-call 确认 |
| 14:35 | IC 指定，开 incident channel |
| 14:40 | 回滚最近部署 |
| 14:45 | 服务恢复 |
| 15:00 | 验证完成，关闭事故 |

## 影响
```text
用户影响：X 用户无法 Y，持续 Z 分钟
业务影响：损失订单 X 笔 / 收入 $Y
数据影响：无 / 有（详述）
```

## 根因分析（5 Whys）
```text
Why 1：订单创建失败
Why 2：数据库连接超时
Why 3：连接池耗尽
Why 4：新部署的代码有连接泄漏
Why 5：Code Review 未检查连接关闭

根因：代码连接泄漏 + Review 未覆盖
```

## 触发条件
```text
直接触发：部署了含连接泄漏的代码
为什么没提前发现：
  - 单元测试未覆盖连接关闭
  - staging 数据量小未触发
  - 连接池监控告警阈值过高
```

## 做得好的
```text
- 5 分钟内确认
- 回滚决策快
- 通知及时
```

## 做得不好的
```text
- 连接池监控阈值太高（90% 才告警）
- staging 数据量不够真实
- Code Review 未关注资源释放
```

## 改进项（Action Items）
| # | 改进项 | 负责人 | 截止 | 状态 |
|---|-------|-------|------|------|
| 1 | 连接池告警阈值降到 70% | SRE | 本周 | ⏳ |
| 2 | 加连接泄漏检测（Lint 规则） | Dev | 下周 | ⏳ |
| 3 | staging 数据量提升到生产 50% | DevOps | 2 周 | ⏳ |
| 4 | Code Review 加资源释放检查项 | Tech Lead | 本周 | ⏳ |
