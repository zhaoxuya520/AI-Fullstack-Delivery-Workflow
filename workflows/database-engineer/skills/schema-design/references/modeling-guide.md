# 数据建模指南

## 建模顺序

1. 从业务目标识别核心实体。
2. 标注实体关系和基数：一对一、一对多、多对多。
3. 明确实体生命周期、状态流转、删除和归档策略。
4. 把必须成立的业务不变量落到约束。
5. 再根据查询场景补充必要冗余。

## 表结构原则

- 表名表达业务实体或关系，不表达页面。
- 字段名表达稳定语义，不表达临时 UI 文案。
- 核心事实使用结构化字段，不滥用 JSON。
- NULL、默认值、枚举必须有明确语义。
- 审计字段和租户字段应在设计阶段明确。

## 约束原则

- 主键用于稳定标识。
- 唯一约束用于兜住业务唯一性。
- 外键是否启用要结合数据库、写入压力和治理方式，但关系必须文档化。
- 检查约束适合表达简单、稳定、数据库可验证的规则。
