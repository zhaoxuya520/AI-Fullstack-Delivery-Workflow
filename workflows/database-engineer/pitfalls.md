# 数据库工程师工作流常见坑

## 1. 只按页面字段建表

- 表现：页面有什么字段就建什么列，忽略业务实体、生命周期和关系。
- 风险：后续 API、权限、统计、归档需求一出现就需要大改 schema。
- 避免方式：先识别实体、关系、状态流转和查询场景，再落字段。

## 2. 缺少唯一约束导致脏数据

- 表现：只在业务代码里判断唯一，不在数据库层建立唯一约束。
- 风险：并发写入、重试、导入脚本会产生重复记录。
- 避免方式：把业务必须唯一的事实落到唯一索引或唯一约束中。

## 3. 把索引当性能万能药

- 表现：看到查询字段就加索引，或者给每个字段都加索引。
- 风险：写入变慢、磁盘膨胀、优化器选择变差，迁移时间变长。
- 避免方式：从真实访问模式、过滤选择性、排序分页、读写比例反推索引。

## 4. 迁移只考虑新环境

- 表现：DDL 在空库可执行，但没有考虑生产存量数据、锁表、默认值、回填时间。
- 风险：上线失败、长事务、锁表、数据不一致。
- 避免方式：拆分为兼容字段新增、回填、校验、切换、清理阶段。

## 5. 没有回滚和备份方案

- 表现：只给 forward migration，不说明失败后如何恢复。
- 风险：生产变更一旦失败只能人工抢修。
- 避免方式：每个迁移计划同时输出回滚条件、备份点、恢复验证步骤。

## 6. 只审单条 SQL，不看整体读写模式

- 表现：只优化某条 SQL，忽略调用频率、分页方式、事务范围、N+1。
- 风险：局部优化掩盖系统性瓶颈。
- 避免方式：把 SQL 放回业务路径，建立“查询场景 → SQL → 索引 → 验证指标”的映射。

## 7. 多租户隔离边界不清

- 表现：有 `tenant_id` 字段但没有查询约束、唯一约束、权限边界和测试策略。
- 风险：越权读取、跨租户污染、审计失败。
- 避免方式：明确租户键、复合唯一约束、默认过滤策略、后台操作边界和测试用例。

## 8. JSON / 大字段使用失控

- 表现：把频繁查询、筛选、排序的字段塞进 JSON 或大文本。
- 风险：难索引、难校验、难迁移、难统计。
- 避免方式：只有低频、弱结构、展示型扩展信息才进入 JSON；核心查询字段应结构化。

## 9. 忽略数据生命周期

- 表现：只设计新增和查询，不考虑软删、归档、保留期、审计、恢复。
- 风险：数据膨胀、合规风险、误删不可恢复。
- 避免方式：在 schema 设计阶段明确删除策略、归档策略和审计字段。

## 10. DDL、后端模型、API 语义不一致

- 表现：数据库字段名、API 字段名、ORM 模型含义不一致。
- 风险：联调反复返工，数据含义被误用。
- 避免方式：交付时同步字段语义、约束、默认值、枚举和空值规则。
