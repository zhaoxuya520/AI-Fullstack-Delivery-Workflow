# 产品经理工作流公开资料索引

本索引用于记录产品经理工作流参考的公开资料来源。引用公开资料时只沉淀方法、结构和检查点，不复制受版权保护的大段原文。

## PRD / 产品需求

| 来源 | 主题 | 可复用要点 |
|---|---|---|
| [Atlassian: Product Requirements Document](https://www.atlassian.com/agile/product-management/requirements) | PRD | 需求应连接目标、用户需求、功能、成功标准和协作过程。 |
| [Atlassian PRD Template](https://www.atlassian.com/software/confluence/templates/product-requirements?category=services) | PRD 模板 | PRD 应作为跨团队共享文档，而不是单人静态文档。 |
| [Atlassian: Product Specifications](https://www.atlassian.com/agile/product-management/product-specification) | 产品规格 | 规格文档应说明范围、用户故事和需求边界。 |

## 用户故事 / 验收标准

| 来源 | 主题 | 可复用要点 |
|---|---|---|
| [Atlassian: User Stories](https://www.atlassian.com/agile/project-management/user-stories) | 用户故事 | 用户故事强调用户、动作和价值，并需要团队共同确认验收标准。 |
| [Atlassian: Acceptance Criteria](https://www.atlassian.com/work-management/project-management/acceptance-criteria) | 验收标准 | 验收标准要清晰、可测试、可验证。 |
| [Agile Alliance: INVEST](https://agilealliance.org/glossary/invest/) | 用户故事质量 | 好故事应 Independent、Negotiable、Valuable、Estimable、Small、Testable。 |
| [Agile Alliance: User Stories](https://agilealliance.org/glossary/user-stories/) | 敏捷需求 | 用户故事是协作入口，不是完整规格替代品。 |
| [Cucumber: Gherkin Reference](https://cucumber.io/docs/gherkin/reference) | Given/When/Then | Gherkin 可用于把业务规则写成可验证场景。 |
| [Martin Fowler: User Story](https://martinfowler.com/bliki/UserStory.html) | 用户故事 | 用户故事是轻量、会话式需求载体。 |
| [Martin Fowler: Story Test](https://martinfowler.com/bliki/StoryTest.html) | 故事测试 | 验收测试帮助确认故事是否真正完成。 |

## 用户需求 / 服务设计 / 发现

| 来源 | 主题 | 可复用要点 |
|---|---|---|
| [GOV.UK: User Needs](https://www.gov.uk/service-manual/user-centred-design/user-needs) | 用户需求 | 需求应从用户需要出发，而不是从组织内部假设出发。 |
| [GOV.UK: Agile Delivery](https://www.gov.uk/service-manual/agile-delivery) | 敏捷交付 | 服务交付应迭代、持续学习，并围绕用户价值组织工作。 |
| [GOV.UK: Writing User Stories](https://www.gov.uk/service-manual/agile-delivery/writing-user-stories/) | 用户故事 | 用户故事应帮助团队理解用户需要和交付价值。 |
| [NN/g: UX Deliverables Glossary](https://www.nngroup.com/articles/ux-deliverables-glossary/) | UX 交付物 | 人物角色、旅程图、服务蓝图等可辅助产品发现和交接。 |
| [NN/g: Customer Journey Map Template](https://media.nngroup.com/media/articles/attachments/JMTemplate.pdf) | 用户旅程 | 旅程图可记录阶段、行为、想法、痛点、机会和负责人。 |
| [SVPG: Product Management Introduction](https://www.svpg.com/product-management-an-introduction/) | 产品发现 | 产品经理要管理价值、可用性、可行性、商业可行性风险。 |
| [SVPG: The Four Big Risks](https://www.svpg.com/four-big-risks/) | 四类风险 | 需求评审应覆盖 value、usability、feasibility、viability。 |
| [SVPG: Product Discovery](https://www.svpg.com/product-discovery/) | 产品发现 | 发现阶段用于验证应该做什么，而不是只写交付清单。 |

## 优先级 / 路线图 / Backlog

| 来源 | 主题 | 可复用要点 |
|---|---|---|
| [Scrum Guide](https://scrumguides.org/scrum-guide.html) | Product Goal / Backlog | Product Owner 负责产品目标、Backlog 条目、排序和透明度。 |
| [Roman Pichler: Product Roadmap and Product Backlog](https://www.romanpichler.com/blog/product-roadmap-product-backlog/) | 路线图与 Backlog | 路线图表达目标和方向，Backlog 承载可执行条目。 |
| [Roman Pichler: Product Strategy Framework](https://www.romanpichler.com/blog/romans-product-strategy-framework-v3/) | 产品战略 | 产品战略应连接愿景、目标市场、需求、差异化和业务目标。 |
| [Roman Pichler: Product Canvas](https://www.romanpichler.com/tools/product-canvas/) | 产品画布 | 可用于早期产品定义和团队对齐。 |
| [ProductPlan: Prioritization Frameworks](https://www.productplan.com/learn/product-management-frameworks) | 优先级框架 | RICE、MoSCoW、Kano、价值/成本等框架适用于不同决策场景。 |
| [ProductPlan: Kano Model](https://www.productplan.com/learn/kano-model-prioritization/) | Kano | Kano 可用于区分基础型、期望型和兴奋型需求。 |
| [Mind the Product: Product Prioritisation 101](https://www.mindtheproduct.com/product-prioritisation-101/) | 优先级 | 优先级应服务于目标和约束，而不是只按个人偏好排序。 |

## 指标 / 结果衡量

| 来源 | 主题 | 可复用要点 |
|---|---|---|
| [Google Research: HEART Framework](https://research.google/pubs/measuring-the-user-experience-on-a-large-scale-user-centered-metrics-for-web-applications/) | HEART 指标 | 可从 Happiness、Engagement、Adoption、Retention、Task Success 选择体验指标。 |
| [Google HEART PDF](https://research.google.com/pubs/archive/36299.pdf) | Goals-Signals-Metrics | 指标设计应从目标到信号再到可量化指标。 |
| [ProductPlan: HEART Framework](https://www.productplan.com/glossary/heart-framework) | 产品指标 | HEART 可作为产品体验度量补充。 |
