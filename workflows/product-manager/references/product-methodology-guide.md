# 产品经理方法论实战指南

> 面向 APP / 小程序 / 网页产品的需求分析、PRD 编写、MVP 规划。

## 1. 需求分析框架

### 用户故事格式

```text
作为 [用户角色]，
我希望 [功能/行为]，
以便 [获得的价值/目标]。

验收标准（Given-When-Then）：
  Given [前置条件]
  When [用户操作]
  Then [预期结果]

示例：
  作为已注册用户，
  我希望通过手机号快捷登录，
  以便不需要记住密码就能访问账户。

  验收标准：
  Given 用户已注册且手机号已验证
  When 用户输入手机号并点击"获取验证码"
  Then 60秒内收到6位验证码短信
  And 验证码60秒内有效
  And 输入正确验证码后成功登录
```

### RICE 优先级评估

```text
R（Reach）: 影响多少用户？（人数/百分比）
I（Impact）: 对每个用户的影响程度？（3=巨大/2=高/1=中/0.5=低/0.25=极低）
C（Confidence）: 有多大把握？（100%=高/80%=中/50%=低）
E（Effort）: 需要多少人天？

RICE Score = (R × I × C) / E

示例：
  功能A: (5000 × 2 × 80%) / 10 = 800 ← 高优
  功能B: (500 × 3 × 50%) / 20 = 37.5 ← 低优
```

## 2. PRD 标准结构

```text
1. 概述
   - 背景与目标
   - 目标用户
   - 核心指标（如何衡量成功）

2. 功能范围
   - In Scope（本期做什么）
   - Out of Scope（本期不做什么）
   - 依赖与约束

3. 用户流程
   - 主流程图（Mermaid）
   - 异常流程

4. 功能详细说明
   - 每个功能点：
     - 描述
     - 用户故事
     - 验收标准
     - 原型链接
     - 优先级（P0/P1/P2）

5. 数据需求
   - 埋点清单
   - 数据报表需求

6. 非功能需求
   - 性能（响应时间/并发）
   - 安全（权限/脱敏）
   - 兼容性（设备/浏览器）

7. 里程碑
   - 开发/联调/测试/上线时间

8. 附录
   - 竞品参考
   - 原始需求来源
```

## 3. MVP 规划方法

```text
┌─ MVP 四象限分析 ─────────────────────────────────────────┐
│                                                           │
│          高用户价值                                       │
│            │                                              │
│    必须做   │  差异化                                     │
│   (P0/MVP) │  (P1/V2)                                   │
│            │                                              │
│  ──────────┼──────────── 低实现成本 ← → 高实现成本       │
│            │                                              │
│    快速做   │  延后做                                     │
│   (P1/V1)  │  (P2/V3)                                   │
│            │                                              │
│          低用户价值                                       │
└───────────────────────────────────────────────────────────┘

MVP 原则：
  - 能验证核心假设的最小功能集
  - 用户能跑通核心流程
  - 不做"nice to have"
  - 不做完美设计（先跑通再优化）
  - 2~4周内可上线

MVP → V1 → V2 节奏：
  MVP（2~4周）: 核心流程跑通
  V1（+2~4周）: 补齐基础体验（错误提示/loading/空状态）
  V2（+4~8周）: 差异化功能 + 数据驱动优化
```

## 4. 竞品分析框架

```text
分析维度：
  1. 产品定位（目标用户 / 核心价值 / 商业模式）
  2. 功能对比（功能矩阵表）
  3. 用户体验（核心流程对比）
  4. 数据表现（下载量/评分/DAU估算）
  5. 差异化机会（他们没做好/没做到的）

输出格式：
  竞品对比矩阵表 + 机会点清单 + 行动建议
```

## 5. 常见坑

```text
1. 需求不写验收标准 → 开发QA理解不一致
2. PRD 只写"参考竞品X" → 不是需求文档
3. 功能范围无边界 → 需求蔓延
4. 不区分优先级 → 全都P0等于没优先级
5. 不量化目标 → 上线不知道算不算成功
6. 过度设计MVP → 3个月做完已错过窗口
7. 不做用户调研就闭门造车
8. 需求变更不走流程 → 团队疲于应对
9. 只写功能不写非功能 → 上线后性能/安全出事
10. 不跟进数据 → 做完不知道效果
```
