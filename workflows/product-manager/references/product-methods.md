# 产品经理工作流方法论参考

## 1. 5W1H 需求澄清

```text
Who：谁使用？
What：要完成什么？
When：什么时候发生？
Where：在哪个场景？
Why：为什么要做？
How：如何完成？
```

适合：模糊需求、客户原话、老板想法。

---

## 2. 用户需求

```text
用户是谁？
用户在什么场景？
用户要完成什么任务？
当前障碍是什么？
如果解决，用户和业务会得到什么价值？
```

适合：产品发现、PRD 背景、服务设计。

---

## 3. 产品发现

```text
产品目标
→ 用户需求
→ 机会点
→ 候选方案
→ 验证实验
→ 决策
```

适合：新产品、高不确定功能、方向不清的需求。

---

## 4. 四类产品风险

```text
价值风险：用户是否需要？
可用性风险：用户是否能用？
可行性风险：技术是否能实现？
商业可行性风险：业务、成本、合规是否允许？
```

适合：需求评审、产品发现、PRD 进入开发前。

---

## 5. MoSCoW 优先级

```text
Must：必须做，没有它产品闭环不成立
Should：应该做，重要但可以后置
Could：可以做，锦上添花
Won't：本阶段不做，明确排除
```

适合：MVP、版本范围、需求砍范围。

---

## 6. RICE 优先级

```text
Reach：影响多少用户
Impact：影响有多大
Confidence：判断有多确定
Effort：成本多高

Score = Reach × Impact × Confidence / Effort
```

适合：多个需求竞争资源时排序。

---

## 7. Kano 模型

```text
基本型需求：没有会不满，有了也不一定惊喜
期望型需求：做得越好越满意
兴奋型需求：用户没预期，但有了会惊喜
无差异需求：做不做影响不大
反向需求：做了反而不喜欢
```

适合：体验功能、用户满意度分析。

---

## 8. 用户故事

```text
作为 <用户角色>
我希望 <完成动作>
以便 <获得价值>
```

重点：不要只写动作，要写价值。

---

## 9. INVEST 用户故事质量

```text
Independent：尽量独立
Negotiable：可协商，不是死规格
Valuable：有用户价值
Estimable：可估算
Small：足够小
Testable：可测试
```

适合：Backlog 梳理、开发前评审。

---

## 10. Given / When / Then

```text
Given 前置条件
When 用户行为
Then 系统结果
```

适合：验收标准、QA 测试用例输入。

---

## 11. HEART 指标

```text
Happiness：满意度
Engagement：参与度
Adoption：采用率
Retention：留存
Task Success：任务成功
```

适合：体验指标、功能上线复盘。

---

## 12. Goals / Signals / Metrics

```text
Goal：目标是什么？
Signal：什么现象说明目标达成？
Metric：如何量化？
```

适合：指标定义、埋点建议、上线复盘。

---

## 13. 用户旅程

```text
触发点
→ 进入路径
→ 核心操作
→ 系统反馈
→ 成功/失败结果
→ 后续行为
```

适合：多页面、多角色、多步骤功能。

---

## 14. 产品路线图

```text
产品目标
→ 阶段主题
→ 主要能力
→ 成功指标
→ 依赖和风险
→ Backlog 候选
```

适合：版本规划、项目经理交接、跨团队对齐。

---

## 15. 需求变更影响分析

每次变更检查：

```text
页面是否变化？
API 是否变化？
数据库是否变化？
权限是否变化？
测试是否变化？
安全风险是否变化？
指标是否变化？
部署是否变化？
文档是否变化？
```
