# 产品指标参考指南

## 1. 指标设计原则

产品指标用于判断功能是否达成目标，而不是为了让报表更好看。

```text
目标 → 信号 → 指标 → 数据来源 → 观察周期 → 决策动作
```

## 2. HEART 框架

参考 Google HEART 框架，可从以下维度选择体验指标：

| 维度 | 含义 | 示例 |
|---|---|---|
| Happiness | 用户满意度和主观感受 | 满意度评分、NPS、反馈情绪 |
| Engagement | 用户参与度 | 使用频次、核心行为次数、停留时长 |
| Adoption | 新用户或新功能采用 | 首次使用率、新功能启用率 |
| Retention | 持续使用 | 次日/7日/30日留存、复购、回访 |
| Task Success | 任务成功 | 完成率、失败率、耗时、错误次数 |

## 3. Goals-Signals-Metrics

| 层级 | 问题 | 示例 |
|---|---|---|
| Goal | 我们想改善什么？ | 用户能更快完成订单提交 |
| Signal | 什么现象说明改善了？ | 提交耗时下降、失败率下降 |
| Metric | 用什么量化？ | 平均提交耗时、提交成功率、错误率 |

## 4. 指标类型

| 类型 | 用途 | 风险 |
|---|---|---|
| 北极星指标 | 统一长期价值方向 | 过度抽象，无法指导短期决策 |
| 输入指标 | 衡量用户行为和过程 | 可能与业务结果弱相关 |
| 输出指标 | 衡量最终结果 | 反馈慢，归因困难 |
| 防护指标 | 防止局部优化伤害整体 | 容易遗漏，需要跨团队讨论 |

## 5. PRD 中的指标写法

```text
目标：减少用户提交订单失败。
信号：用户能顺利完成订单提交。
核心指标：订单提交成功率。
辅助指标：提交耗时、错误提示出现次数、重复提交次数。
防护指标：客服投诉量、退款率、支付异常率。
观察周期：上线后 7 天、14 天、30 天。
成功标准：提交成功率提升到 98% 以上，重复提交次数下降 30%。
```

## 6. 常见错误

```text
只写“提升体验”，没有指标。
只写访问量，不看任务是否成功。
只看转化，不看投诉、错误、性能等防护指标。
指标没有数据来源。
指标没有观察周期。
指标不对应业务目标或用户目标。
```
