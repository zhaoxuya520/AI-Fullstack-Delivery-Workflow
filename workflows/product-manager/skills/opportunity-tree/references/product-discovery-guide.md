# 产品发现参考指南

## 1. 目标

产品发现用于回答：

```text
我们应该解决什么问题？
这个问题是否值得解决？
用户是否真的需要？
方案是否可用、可行、可持续？
```

产品发现不是无限调研，也不是替代交付；它的目标是在投入设计和开发前降低关键不确定性。

---

## 2. 四类产品风险

参考 SVPG 的产品发现思想，需求进入交付前至少检查四类风险：

| 风险 | 核心问题 | 需要谁参与 |
|---|---|---|
| 价值风险 | 用户是否真的需要？是否愿意使用？ | 产品、用户、业务方 |
| 可用性风险 | 用户能否理解并完成任务？ | 产品、UI/UX、用户研究 |
| 可行性风险 | 技术上能否在约束内实现？ | 前端、后端、架构、数据 |
| 商业可行性风险 | 是否符合业务、合规、运营、成本约束？ | 产品、项目、安全、业务负责人 |

## 3. 发现步骤

```text
1. 明确业务目标或产品目标
2. 定义目标用户和使用场景
3. 收集用户问题、行为证据和约束
4. 把问题拆成机会点
5. 为每个机会点提出候选方案
6. 为高不确定方案设计验证实验
7. 根据证据决定：进入交付 / 缩小范围 / 暂缓 / 放弃
```

## 4. 机会-方案-实验结构

```text
产品目标
└─ 用户机会 1
   ├─ 方案 A
   │  └─ 验证实验
   └─ 方案 B
      └─ 验证实验
└─ 用户机会 2
   └─ 方案 C
      └─ 验证实验
```

## 5. 用户需求表达

好的用户需求应避免只写内部视角：

```text
差：系统需要增加批量导出功能。
好：财务人员需要在月末快速导出指定时间范围内的订单数据，以便完成对账。
```

## 6. 发现完成标准

进入交付前检查：

```text
用户是谁？
用户问题是否有证据？
业务目标是否明确？
成功指标是否明确？
核心方案是否经过基本验证？
高风险假设是否列出？
是否明确不做什么？
是否能转成用户故事和验收标准？
```

## 7. 下游交接

产品发现完成后，应交给：

| 下游 | 交付内容 |
|---|---|
| UI/UX | 用户、场景、旅程、痛点、机会点 |
| API/数据库/开发 | 核心业务对象、流程、约束、风险假设 |
| QA | 用户目标、成功路径、失败路径、验收标准 |
| 项目经理 | 优先级、范围、依赖和里程碑建议 |
