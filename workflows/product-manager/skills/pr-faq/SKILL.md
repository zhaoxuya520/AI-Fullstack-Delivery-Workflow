# Amazon PR/FAQ（Working Backwards）

参考来源：[Working Backwards](https://workingbackwards.com/resources/working-backwards-pr-faq/)、[Amazon's Innovation Blueprint](https://productschool.com/blog/product-fundamentals/prfaq)

## 适用场景

- 从零定义新产品
- 大版本/大功能规划
- 团队对"为什么做"有分歧
- 投资人/高层 pitch
- 产品愿景对齐

## 不适用场景

- 小功能迭代（杀鸡用牛刀）
- 已有清晰愿景的项目
- 紧急修复或合规需求

## 核心思想

**从未来的用户体验倒推回来，先写"产品已经发布"的新闻稿。**

```text
传统方式：
  想法 → 业务案例 → 设计 → 开发 → 上线 → 写新闻稿
  问题：开发完才发现用户不需要

Working Backwards：
  写新闻稿（假设已发布）→ 团队讨论是否真有价值 → 不行就改 → 行就开发
  优势：用最低成本验证"用户为什么会用"
```

## PR 结构（一页纸新闻稿）

```markdown
# [标题：一句话说明产品是什么]

## [副标题：目标用户 + 核心价值]

[发布日期] — [公司名] 今天发布了 [产品名]，[一句话产品介绍]。

[第一段：问题段]
[详细描述用户当前遇到什么问题，造成什么损失。]

[第二段：解决方案段]
[产品如何解决这个问题。重点写"用户体验"，不要写技术实现。]

[第三段：用户引言]
"[一个虚构的真实用户用自己的话说他/她的体验]"
— [用户姓名、身份]

[第四段：如何开始]
[用户怎么用上这个产品。访问 URL / 下载 App / 联系销售。]

[第五段：负责人引言]
"[公司高管说为什么我们要做这个，对用户/行业有什么意义]"
— [高管姓名、职位]
```

## FAQ 结构（外部 + 内部）

### 外部 FAQ（用户视角）

```text
□ 这个产品是什么？
□ 谁能用？
□ 怎么用？
□ 多少钱？
□ 和 [竞品 X] 有什么区别？
□ 数据安全吗？
□ 支持哪些平台？
□ 上线时间？
□ 有什么限制？
```

### 内部 FAQ（团队视角）

```text
□ 为什么现在做？
□ 为什么我们做（而不是竞品）？
□ 技术可行吗？
□ 成本多少（开发 + 运营）？
□ 有什么风险？
□ 需要哪些资源？
□ 失败标准是什么？
□ 成功指标是什么？
□ 对现有产品有什么影响？
□ 法务/合规考虑？
```

## 工作流程

```text
1. 写 PR 第一版（不超过一页）
   ↓
2. 团队评审 → 大家是否被这个产品打动？
   ├─ 否 → 修改 PR 或放弃
   └─ 是 → 继续
   ↓
3. 写外部 FAQ（10~15 条）
   ↓
4. 写内部 FAQ（10~15 条，更深入）
   ↓
5. 内部 FAQ 暴露关键风险和未决问题
   ↓
6. 决策：投入开发 / 暂缓 / 放弃
   ↓
7. 投入开发 → PR/FAQ 成为团队的北极星文档
```

## 写 PR 的关键原则

### 1. 用用户语言，不用技术语言

```text
❌ "采用先进的 AI 推荐算法"
✅ "用户打开 App 第一眼就能看到他们想要的"
```

### 2. 重点写"为什么"和"做什么"，不写"怎么做"

```text
❌ "通过深度学习模型分析用户行为..."
✅ "新用户第 1 天就能找到 3 个感兴趣的话题"
```

### 3. 引言要可信

```text
❌ "这个产品太棒了！" — 用户
✅ "我以前每周花 8 小时手动整理报表，现在用这个工具 10 分钟就完成了。" — 王经理，某 SaaS 公司财务总监
```

### 4. 避免"全球第一""革命性"等空话

```text
❌ "颠覆性的 AI 解决方案"
✅ "把搜索准确率从 60% 提升到 90%"
```

## 完整示例

```markdown
# Notion AI：把笔记变成你的第二大脑

## 让每个写作者都拥有 AI 写作助手

2023 年 2 月 22 日 — Notion 今天发布了 Notion AI，第一个完全集成在笔记工具中的 AI 写作助手。用户无需切换工具，即可在写作过程中获得 AI 辅助。

**问题**

每天有数百万人使用笔记工具来思考、写作和协作。但当他们想用 AI 提升写作效率时，必须打开另一个标签页（如 ChatGPT），把内容复制过去，等待结果，再复制回来。这个过程不仅繁琐，还打断了思考流。更糟糕的是，AI 生成的内容缺少笔记的上下文，质量打了折扣。

**解决方案**

Notion AI 内置在每个 Notion 页面中。用户在写作时按下空格键即可调出 AI，让它续写、总结、翻译、改写或生成大纲。AI 可以读取当前页面的上下文，给出贴合主题的回答。所有操作不离开 Notion，写作流程不被打断。

**用户体验**

"我每周写 5 篇博客，以前每篇要 3 小时。用了 Notion AI 之后，AI 帮我快速搭建大纲、扩展观点、检查语法，我只需要 1 小时就能完成。" — Sarah，独立内容创作者

**如何开始**

所有 Notion 用户立即可用，每月免费 20 次 AI 调用，付费版无限次。访问 notion.so/ai 了解详情。

**负责人引言**

"我们相信 AI 不应该是另一个工具，而应该融入你已经在用的工具中。Notion AI 让 AI 写作助手变得像拼写检查一样自然。" — Ivan Zhao，Notion CEO
```

## 质量自检

```text
□ PR 是否一页纸（不超过 600 字）
□ 用户语言而非技术语言
□ 引言是否可信（具体场景、可量化结果）
□ 是否避免了"革命性""颠覆"等空话
□ 外部 FAQ 是否覆盖用户最关心的 10+ 问题
□ 内部 FAQ 是否暴露了真实风险
□ 团队读完是否有"我想要这个产品"的感觉
```

## 常见坑

1. **PR 写成产品文档**——堆砌功能而不是讲故事
2. **用户引言假**——"这个产品改变了我的生活！"（没人这么说话）
3. **没写问题段**——直接讲解决方案
4. **内部 FAQ 避重就轻**——不敢写真实风险
5. **PR 完了就放着**——应该作为团队的北极星文档持续引用
6. **太多技术细节**——PR 是给非技术人员看的
7. **过度承诺**——写得像不存在的完美产品

