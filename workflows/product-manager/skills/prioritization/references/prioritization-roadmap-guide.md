# 优先级和路线图参考指南

## 1. 优先级不是投票

优先级用于在目标、用户价值、风险、成本和时间之间做取舍。不要只按声音最大的人、职位最高的人或个人偏好排序。

## 2. 常用优先级方法

| 方法 | 适合场景 | 不适合场景 |
|---|---|---|
| MoSCoW | MVP 切分、版本边界 | 所有需求都被写成 Must 时失效 |
| RICE | 多功能定量排序 | 没有 Reach/Impact/Confidence 数据时容易伪精确 |
| Kano | 体验满意度分类 | 技术平台、合规需求不一定适合 |
| 价值/成本矩阵 | 快速评审和粗排 | 复杂依赖项目容易过度简化 |
| 风险优先 | 高不确定性项目 | 已明确交付清单时可能拖慢推进 |

## 3. MoSCoW 判断标准

```text
Must：没有它，产品闭环不成立或无法上线。
Should：重要，但缺少它仍可形成 MVP。
Could：有价值，但可作为锦上添花。
Won't：本阶段明确不做，防止范围膨胀。
```

## 4. RICE 使用提醒

```text
Reach：影响多少用户或业务对象。
Impact：影响程度。
Confidence：判断把握。
Effort：实现成本。
Score = Reach × Impact × Confidence / Effort
```

使用 RICE 时必须记录估算来源，否则分数容易变成主观包装。

## 5. 路线图和 Backlog 区别

| 项目 | 关注点 | 粒度 |
|---|---|---|
| 产品目标 | 为什么做 | 战略和结果 |
| 路线图 | 先解决哪些目标和主题 | 阶段、主题、版本 |
| Backlog | 具体要做哪些条目 | 用户故事、任务、缺陷 |

路线图不应被写成不可变化的承诺清单；它应表达目标、优先级和阶段性判断，并随着证据更新。

## 6. 路线图条目建议字段

```text
目标：
用户/场景：
要解决的问题：
候选功能：
成功指标：
优先级：
依赖：
风险：
目标版本：
不做范围：
```

## 7. 评审问题

```text
这个条目服务哪个产品目标？
影响哪些用户？
为什么现在做？
如果不做会怎样？
是否有更小的验证版本？
依赖哪些团队或系统？
是否有安全、合规或数据风险？
成功指标是什么？
```
