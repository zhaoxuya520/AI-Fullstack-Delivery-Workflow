# 产品定位声明（Positioning Statement）

参考来源：Geoffrey Moore《Crossing the Chasm》

## 适用场景

- 新产品从零定义
- 团队对"我们到底在做什么"有分歧
- 需要向投资人/客户/合作方解释产品
- 营销文案的源头
- 产品路线图的方向锚点

## 不适用场景

- 已有定位的产品（除非要 pivot）
- 单功能迭代（不需要重新定位）
- 内部工具（不对外部讲）

## 核心模板

```text
对于 [目标用户]
他们 [有什么需求/痛点]
[产品名] 是一个 [产品类别]
它能 [关键价值/差异化]
不同于 [竞品/替代方案]
我们的产品 [独特优势]
```

## 六个要素详解

### 1. 目标用户（For）

具体到能想象出真人，不要模糊群体。

```text
❌ "对于所有想提升效率的人"
✅ "对于 5~50 人的小型设计团队的项目经理"
```

### 2. 需求/痛点（Who）

用用户语言描述他们当前的问题。

```text
❌ "他们需要更好的工具"
✅ "他们每周花 10+ 小时手动跟踪设计评审进度"
```

### 3. 产品类别（Is a）

用一句话说出产品是什么类型。让用户能立即归类。

```text
❌ "一个革命性的 AI 平台"
✅ "一个面向设计团队的协作工具"
```

### 4. 关键价值（That）

最核心的 1~2 个价值点。

```text
❌ "提供完整的解决方案"
✅ "自动跟踪每个设计稿的评审状态，节省 8 小时/周"
```

### 5. 竞品/替代方案（Unlike）

明确指出和谁比。**没有竞品就是没有市场**。

```text
✅ "不同于 Figma 的评论功能"
✅ "不同于用 Notion 手动维护评审表"
✅ "不同于发邮件追问每个 reviewer"
```

### 6. 独特优势（Our product）

为什么选你而不是竞品。

```text
❌ "更好用、更快"
✅ "自动化评审流程 + 与 Figma 双向同步 + 历史评审记录可追溯"
```

## 工作示例

```text
对于 5~50 人小型设计团队的项目经理（PM）
他们每周花 10+ 小时手动跟踪设计评审进度
ReviewFlow 是一个面向设计团队的协作工具
它能自动跟踪每个设计稿的评审状态，节省 8 小时/周
不同于 Figma 内置评论或用 Notion 手动维护评审表
我们的产品提供自动化评审流程 + 与 Figma 双向同步 + 历史评审记录可追溯
```

## 工作流程

```text
1. 列出当前对产品的所有理解（团队各人版本）
   ↓
2. 找出冲突点和共识点
   ↓
3. 用六要素模板填空
   ↓
4. 每个空格至少有 3 个候选 → 选最具体的
   ↓
5. 整合成一段话
   ↓
6. 用以下问题压力测试：
   - 把"产品名"换成竞品，是否还成立？
   - 用户能不能立刻理解？
   - 团队成员能不能用同一句话复述？
   ↓
7. 输出最终定位声明
```

## 质量自检

```text
□ 目标用户是否具体（能想象出真人）
□ 痛点是否用用户语言（不是解决方案语言）
□ 产品类别是否清晰（用户能立即归类）
□ 关键价值是否可量化（不是"更好"而是"X% 提升"）
□ 是否明确指出了竞品（包括"不用任何工具"也是替代方案）
□ 独特优势是否真的独特（竞品也能说同样的话吗？）
□ 整体读起来是否自然（不是堆砌买点）
```

## 常见坑

1. **目标用户太宽**——"所有用户" → 没有用户
2. **痛点是解决方案**——"需要 AI 推荐" → 应该是"找不到想要的内容"
3. **没有竞品**——团队认为"我们是独一无二的" → 用户在用 Excel 也是替代方案
4. **价值不可量化**——"更高效" → "节省 8 小时/周"
5. **优势是功能**——"提供 100+ 模板" → "为什么对用户重要？"
6. **太长**——超过 5 句话就该拆
7. **从内向外说**——讲技术不讲价值

## 测试方法

### 替换测试

把产品名替换成主要竞品，看声明是否还成立。

```text
原版：
"对于设计团队的 PM... ReviewFlow 是一个协作工具..."

替换为 Figma：
"对于设计团队的 PM... Figma 是一个协作工具..."

→ 还成立？说明定位不够独特，需要细化
```

### 5 秒测试

把声明给一个不了解产品的人看 5 秒，问：
- 这是给谁用的？
- 解决什么问题？
- 和别的工具有什么区别？

如果答不上来，说明定位不够清晰。
