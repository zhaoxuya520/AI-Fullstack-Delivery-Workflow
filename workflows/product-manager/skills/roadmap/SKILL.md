# 产品路线图（Roadmap）

## 适用场景

- 季度/年度规划
- 多功能的前后顺序对齐
- 给团队/客户/投资人的方向展示
- Backlog 太多需要收敛

## 不适用场景

- 单功能开发（用 PRD）
- 探索阶段（用 OST）
- 紧急修复（无需路线图）

## 核心原则

```text
1. 主题驱动，不是功能清单
   ❌ Q1: 功能 A、功能 B、功能 C
   ✅ Q1: 提升新用户激活率（包含功能 A、B、C）

2. 路线图不是承诺清单
   随着证据和约束变化而调整

3. 先有目标，再有阶段
   不能先列功能再凑主题

4. 标注不确定性
   越远的阶段越模糊，留出调整空间

5. 每个阶段有成功指标
   不能"做完就算成功"
```

## Now-Next-Later 路线图（推荐）

参考来源：[Janna Bastow](https://www.prodpad.com/blog/no-dates-roadmap/)

```markdown
## 产品路线图

### Now（正在做，0~6 周）
**主题：提升新用户激活率**
- 目标：D7 激活率从 30% 提升到 45%
- 包含：注册流程优化、新手引导、首次价值瞬间

### Next（接下来，6~12 周）
**主题：深化核心场景的使用粘性**
- 目标：DAU/MAU 从 25% 提升到 35%
- 候选方向：通知系统、协作功能、个性化推荐

### Later（更远，12 周后）
**主题：扩展用户场景**
- 目标：客单价 LTV 提升 30%
- 候选方向：付费功能、企业版、API 开放

### Won't（明确不做）
- ❌ 移动端原生 APP（仅 H5）
- ❌ 多语言（仅中文）
- ❌ 直播功能
```

## OKR 风格路线图（季度规划）

参考来源：[Google OKR Playbook](https://www.whatmatters.com/resources/google-okr-playbook)

```markdown
## Q1 2026 路线图

### Objective 1: 提升新用户激活率
**为什么**：D7 激活率仅 30%，远低于行业平均

| Key Result | 当前 | 目标 |
|-----------|------|------|
| KR1: D7 激活率 | 30% | 45% |
| KR2: 新用户首次完成核心操作 | 40% | 70% |
| KR3: 新手引导完成率 | 25% | 60% |

**行动项**：
- E1: 注册流程优化（M 级，2h）
- E2: 新手引导改造（L 级，4h）
- E3: 首次价值瞬间设计（M 级，1.5h）

### Objective 2: 深化使用粘性
...
```

## 主题选择方法

### 1. 自上而下：从 OKR 推导

```text
公司 OKR
  → 产品 OKR
    → 路线图主题

例：
公司 OKR：年营收增长 50%
  → 产品 OKR：付费用户从 10K 到 20K
    → 路线图主题：
      - 提升试用→付费转化率
      - 降低付费用户流失
      - 拓展付费场景
```

### 2. 自下而上：从用户洞察推导

```text
用户研究 / 数据分析
  → 识别核心痛点
    → 路线图主题

例：
用户访谈：30% 用户提到"不知道这个产品能干啥"
数据：D1 跳出率 60%
  → 主题：让用户在第一天理解产品价值
```

### 3. 战略驱动：从市场机会推导

```text
市场分析
  → 识别战略机会
    → 路线图主题

例：
竞品分析：竞品都没做 AI 智能助手
市场调研：用户对 AI 接受度高
  → 主题：用 AI 重新定义核心交互
```

## 工作流程

```text
1. 选择路线图风格（Now-Next-Later / OKR）
   ↓
2. 收集输入：
   - 公司/产品 OKR
   - 用户洞察（来自 customer-journey, opportunity-tree）
   - 数据分析
   - 战略方向（来自 pr-faq, positioning）
   ↓
3. 提炼 3~5 个主题（不要超过 5 个）
   ↓
4. 每个主题定义：
   - 为什么（背景）
   - 成功指标
   - 候选行动项
   ↓
5. 标注不确定性（确定 / 候选 / 探索）
   ↓
6. 列出 Won't（明确不做）
   ↓
7. 输出路线图
   ↓
8. 季度复盘 + 调整
```

## 路线图输出模板

```markdown
# [产品名] 路线图 [周期]

## 战略背景

[一段话描述当前阶段的核心目标和挑战]

## Now（0~6 周）

### 主题 1: [主题名]

**为什么**：[业务/用户/数据驱动的原因]

**成功指标**：
- 指标 1: 从 X 到 Y
- 指标 2: 从 A 到 B

**行动项**：
- 已确定：[功能 A] [功能 B]
- 候选：[功能 C]
- 探索：[功能 D]

**风险**：
- [可能影响完成的风险]

### 主题 2: [主题名]
...

## Next（6~12 周）

[较粗的描述，主题级]

## Later（12 周后）

[更粗的描述，方向级]

## Won't（明确不做）

- ❌ [明确排除项 + 原因]

## 调整记录

| 日期 | 调整内容 | 原因 |
|------|---------|------|
| ... | ... | ... |
```

## 质量自检

```text
□ 主题数量是否控制在 3~5 个
□ 每个主题是否有成功指标（可量化）
□ 是否标注了不确定性（确定/候选/探索）
□ 是否有 Won't 清单
□ Now 是否详细，Later 是否粗略（越远越模糊）
□ 是否能对应到 OKR 或战略目标
□ 是否有调整记录机制（路线图不是一成不变的）
```

## 常见坑

1. **路线图变成功能清单**——失去了"主题"的指导意义
2. **过度详细**——半年后的事情排得清清楚楚（不可能准确）
3. **没有成功指标**——做完不知道是否成功
4. **不接受调整**——把路线图当成承诺，不愿意根据证据更新
5. **没有 Won't**——结果被各种"加塞"打乱
6. **主题太多**——10 个主题 = 没有主题
7. **闭门造车**——不基于用户洞察和数据

## 路线图沟通原则

```text
对内（团队）：
  - 强调主题和目标
  - 留出执行细节空间
  - 明确不确定性

对外（客户/投资人）：
  - 强调战略方向
  - 不承诺具体时间
  - 用主题包装功能

对销售：
  - 区分"已上线"/"开发中"/"探索中"
  - 不承诺未上线的功能
```

## 与其他 skill 的协作

```text
上游：
  pr-faq → 提供愿景
  positioning → 提供产品定位
  opportunity-tree → 提供机会候选
  prioritization → 提供候选方案的优先级
  customer-journey → 提供用户痛点

下游：
  转交项目经理 → 拆季度 / Sprint 计划
  转交各执行 skill（PRD/MVP/User Story） → 输出具体需求
```
