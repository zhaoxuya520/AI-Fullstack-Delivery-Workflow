## 按任务类型路由

| 用户任务 | 本工作流动作 | 输出 |
|---|---|---|
| “我想做一个项目” | 澄清目标，拆 MVP | MVP 范围、功能清单、风险 |
| “帮我写 PRD” | 补齐背景、功能、规则、验收 | PRD |
| “这个想法靠不靠谱” | 做产品发现和风险检查 | 发现 Brief、关键假设、验证计划 |
| “用户到底需要什么” | 梳理用户需求和场景 | 用户需求、用户画像、用户旅程 |
| “帮我拆需求” | 拆用户故事和功能模块 | 用户故事、任务包 |
| “第一版做什么” | 划分 Must/Should/Could/Won't | MVP 和版本边界 |
| “这些功能怎么排序” | 选择 MoSCoW / RICE / Kano / 价值成本方法 | 优先级列表 |
| “这个功能怎么验收” | 写 Given/When/Then | 验收标准 |
| “这个功能怎么衡量效果” | 定义目标、信号、指标和埋点 | 指标方案、复盘计划 |
| “帮我做路线图” | 对齐产品目标、阶段、依赖和 Backlog | 路线图 |
| “需求变更影响什么” | 分析上下游影响 | 影响范围、风险、下游工作流 |
| “竞品这个功能怎么做” | 抽象用户价值和功能结构 | 竞品分析、可借鉴点 |

## 进入前补问规则

如果缺少目标用户，先问：

```text
这个功能主要给谁用？管理员、普通用户、商家、内部运营，还是其他角色？
```

如果缺少业务目标，先问：

```text
这个功能主要想提升什么：效率、转化、留存、收入、风控、体验，还是降低成本？
```

如果缺少输出类型，先问：

```text
你希望我输出 PRD、MVP 范围、用户故事、验收标准、指标方案、路线图，还是开发任务包？
```

如果缺少范围边界，先问：

```text
第一版必须完成什么？哪些功能可以后续版本再做？
```

如果缺少证据来源，先问：

```text
这个判断来自用户反馈、数据、竞品、业务方要求，还是当前假设？
```

## 转出规则

| 条件 | 转出到 |
|---|---|
| 页面和交互已明确，需要设计 | UI/UX 设计工作流 |
| 需要接口契约 | API 设计工作流 |
| 需要数据实体和表结构 | 数据库工程师工作流 |
| 需要具体开发 | 前端 / 后端 / 全栈工作流 |
| 需要测试用例 | QA 工作流 |
| 涉及权限、敏感数据、攻击面 | 安全工程师工作流 |
| 需要指标口径、埋点分析或复盘看板 | 数据分析工作流 |
| 需要排期和里程碑 | 项目经理工作流 |
| 需要正式文档整理 | 技术文档工作流 |

## 路由未命中

如果任务主要是代码、部署、测试、安全验证或数据分析，不应强行进入产品经理工作流，应返回根 `routing.md` 重新选择。
