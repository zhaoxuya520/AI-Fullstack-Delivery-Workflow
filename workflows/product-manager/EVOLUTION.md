# 产品经理工作流自进化协议

## 1. 目标

产品经理工作流的自进化目标是：

```text
让每一次需求分析、PRD 编写、MVP 拆解、用户故事和验收标准设计，都能沉淀为下一次可复用的模板、规则、路由或经验。
```

---

## 2. 自进化对象

产品经理工作流可以更新：

```text
WORKFLOW.md                 # 产品工作流主流程
routing.md                  # 需求类任务路由
pitfalls.md                 # 需求常见坑
tool-index.md               # 产品方法和模板索引
templates/                  # PRD、MVP、用户故事、验收标准模板
references/                 # 产品方法论、优先级模型、需求评审规范
scripts/                    # PRD 检查、模板生成等自动化脚本
field-journal/              # 真实项目需求经验
```

影响多个岗位时，同步根级：

```text
../../routing.md
../../RULES.md
../../workflow-map.md
../../tool-index.md
../../field-journal/_index.md
```

---

## 3. 任务完成后 Checklist

每次完成产品经理任务后检查：

```text
1. 是否出现新的需求类型？
   → 更新 routing.md

2. 是否有新的产品文档格式？
   → 放入 templates/

3. 是否发现新的需求坑？
   → 更新 pitfalls.md

4. 是否使用了新的产品方法？
   → 更新 tool-index.md 或 references/

5. 是否形成新的下游交接方式？
   → 更新 WORKFLOW.md 和 workflow-map.md

6. 是否有可复用案例？
   → 写入 field-journal/

7. 是否形成跨工作流规则？
   → 更新根 RULES.md
```

---

## 4. 值得沉淀的内容

应该沉淀：

```text
某类项目的 MVP 切分方式
某类功能的 PRD 模板
需求评审中发现的高频问题
用户故事拆解模式
验收标准写法
需求变更影响分析方法
产品到设计/API/数据库/QA 的交接清单
```

不应该沉淀：

```text
一次性的客户原话
没有验证的产品猜想
会议流水账
敏感商业信息
不可复用的临时排期
```

---

## 5. Field Journal 写入标准

当一次任务产生可复用经验时，在 `field-journal/` 新增记录：

```text
日期
项目类型
需求背景
产品问题
采用的拆解方式
最终输出
验收方式
可复用经验
后续改进
标签
```

同时更新：

```text
field-journal/_index.md
```

---

## 6. 产品经理工作流升级标准

只有满足以下条件之一，才更新主流程：

```text
连续多次任务出现同类需求场景
现有模板无法覆盖新需求类型
下游工作流交接出现重复问题
某个检查项能显著减少返工
某个路由关键词频繁命中但未记录
```

---

## 7. 与逆向/渗透工作流的差异

逆向/渗透工作流更重视工具、证据、报告和安全授权。

产品经理工作流更重视：

```text
问题定义
范围边界
验收标准
下游交接
需求变更影响
```

但两者自进化机制一致：

```text
任务完成
→ 生成文档
→ 检查是否可复用
→ 更新路由/工具/模板/坑位/经验
```
