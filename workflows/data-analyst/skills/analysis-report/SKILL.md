# 分析报告（Analysis Report）

## 适用场景

- 业务数据周报 / 月报
- 专项分析（流失 / 转化 / 增长）
- A/B 测试结论报告
- 决策支持报告
- 数据驱动 PRD

## 核心原则

```text
1. 报告给决策者看
   首页有结论和建议

2. 洞察 > 数字
   不只列数字，要解释"为什么"和"怎么办"

3. So What？
   每个发现要回答"所以呢？"

4. 建议可行动
   不是"加强运营"，是"对流失用户发 7 天优惠券"

5. 数据有出处
   可复现（附 SQL / 链接）
```

## 报告结构

```text
1. 摘要（30 秒能看完）
2. 背景（为什么做这个分析）
3. 关键发现（3~5 条）
4. 详细分析（图表 + 解读）
5. 建议与行动项
6. 局限性（诚实）
7. 附录（SQL / 数据）
```

## 工作流程

```text
1. 明确分析目标（与 PM 对齐）
2. 数据收集（SQL / 日志）
3. 分析（漏斗 / 留存 / 对比）
4. 可视化（图表）
5. 提炼洞察（So What？）
6. 写建议（可行动）
7. 评审（与 PM / 业务方）
8. 沉淀（field-journal）
```

## 质量自检

```text
□ 摘要有结论
□ 发现有数据支撑
□ 洞察回答 So What
□ 建议可行动
□ 图表配合文字
□ 局限性诚实
□ SQL 可复现
□ 与决策者评审过
```

## 常见坑

1. **只列数字不解释**——决策者看不懂
2. **建议太宽泛**——"提升留存"不可行动
3. **数据不可复现**——无 SQL 链接
4. **不说局限性**——过度自信
5. **报告太长**——没人看
