# 数据分析师工作流（Data Analyst Workflow）

## 定位

数据分析师工作流负责把业务问题、产品假设、运营目标转化为 **可量化、可验证、可行动** 的数据洞察、指标体系、实验结论和分析报告。

它不替代数据工程师工作流（ETL / 数据仓库）、后端工程师工作流（数据接口实现）、产品经理工作流（需求定义），而是负责 **用数据回答业务问题、验证假设、驱动决策**。

本工作流采用 **skills 模块化架构**：总控负责路由和通用规则，具体方法论拆分成独立 skills，按需加载。

---

## 适用场景

```text
指标体系设计（北极星 / AARRR / 漏斗 / GSM）
SQL 数据查询与分析（聚合 / 窗口函数 / CTE / 子查询）
A/B 测试设计与分析（假设 / 样本量 / 置信度 / 结论）
数据可视化（图表选型 / 仪表盘设计 / 工具选型）
分析报告撰写（洞察 + 建议 + 行动项）
用户行为分析（留存 / 转化 / 路径 / 分群）
业务指标监控与异常归因
漏斗分析 / 归因分析 / 同期群分析
数据口径定义与对齐
数据驱动决策支持
```

## 不适用场景

| 任务 | 应转交工作流 |
|---|---|
| 需求不清、业务目标未定义 | 产品经理工作流 |
| ETL 管道、数据仓库建设、数据治理 | 数据工程师 / 数据库工程师工作流 |
| 机器学习模型训练、特征工程 | AI/ML 工程师工作流 |
| 后端接口开发、数据 API | 后端工程师工作流 |
| 前端页面开发、图表组件实现 | 前端工程师工作流 |
| 项目排期、资源分配 | 项目经理工作流 |
| 数据安全、脱敏、合规审计 | 安全工程师工作流 |
| 线上监控告警、SLO | SRE/运维工作流 |
| UI/仪表盘视觉设计规范 | UI/UX 设计工作流 |

## 输入

### 必需输入

```text
业务问题或分析目标
数据来源（数据库 / 数据仓库 / 埋点 / 日志）
时间范围
目标受众（给谁看：CEO / PM / 运营 / 开发）
输出格式期望（报告 / 仪表盘 / PPT / SQL 脚本）
```

### 可选输入

```text
已有指标定义文档
数据字典 / 表结构说明
历史分析报告
业务假设或待验证命题
产品版本变更记录
竞品数据（如有）
SLA / 性能基线
```

### 输入不足时先补问

```text
1. 分析目标是什么？要回答什么业务问题？
2. 数据来源在哪？有数据字典吗？
3. 时间范围多长？需要同比 / 环比吗？
4. 报告给谁看？技术 or 业务受众？
5. 是验证假设（A/B 测试）还是探索发现？
6. 指标口径是否已对齐？有已有标准吗？
7. 数据质量如何？有已知缺失 / 异常吗？
8. 是否有行动权限约束（只分析 vs 可建议 vs 可决策）？
```

## 完整行为链（硬性流程）

```text
1. 读取分析目标 / 业务问题 / 数据来源 / 受众
   ↓
2. 检查 field-journal/_index.md → 是否有同类分析经验可复用
   ↓
3. 读取 skills/routing.md → 路由到需要的 skills
   ↓
4. 判断分析复杂度（S/M/L/XL）→ 选择产出粒度
   ↓
5. 明确数据口径 → 对齐指标定义
   ↓
6. 加载命中 skills → 按 skill 内方法执行分析
   ↓
7. 数据验证 → 交叉检验 / 抽样核查 / 逻辑自洽
   ↓
8. 输出分析报告 / 仪表盘 / SQL / 结论
   ↓
9. 给出行动建议 + 明确下一步
   ↓
10. 转交产品 / 运营 / 开发 / 决策者
    ↓
11. 按 EVOLUTION.md 沉淀经验 → 回写 field-journal
```

## 禁止行为

```text
❌ 不要在业务目标不清时直接写 SQL（会做无用分析）
❌ 不要跳过数据口径对齐（同一指标各方理解不同 = 灾难）
❌ 不要只给数据不给结论（数字本身不是洞察）
❌ 不要用相关性冒充因果性
❌ 不要忽略数据质量问题（缺失 / 重复 / 异常值）
❌ 不要用误导性图表（截断 Y 轴 / 错误图表类型）
❌ 不要在样本量不足时得出结论
❌ 不要忽略 Simpson 悖论和混淆变量
❌ 不要让分析报告只有"发现"没有"建议"
❌ 不要把一次性查询当永久指标（需沉淀口径）
❌ 不要跳过数据验证直接输出结论
❌ 不要忽略受众差异（给 CEO 和给开发的报告不同）
```

## 任务复杂度分级

```text
S 级（10~30 分钟）：单指标查询 / 单图表 / 单次 SQL
  → sql-analysis

M 级（30~120 分钟）：单主题分析（留存 / 转化 / 漏斗）
  → metric-design + sql-analysis + data-visualization

L 级（2~6 小时）：A/B 测试分析 / 多维度专题报告
  → metric-design + sql-analysis + ab-testing + data-visualization + analysis-report

XL 级（6 小时+）：全业务指标体系设计 / 季度数据复盘
  → 全部 5 个 skills + 多轮迭代
```

## 通用质量检查

```text
□ 分析目标是否明确？回答了什么业务问题？
□ 指标口径是否已对齐？定义是否无歧义？
□ 数据来源是否可靠？是否确认数据质量？
□ 时间范围是否合理？是否排除异常时段？
□ 是否做了数据验证（交叉检验 / 抽样核查）？
□ 结论是否有数据支撑？是否区分相关与因果？
□ 是否考虑了混淆变量 / 幸存者偏差 / Simpson 悖论？
□ 可视化是否准确、不误导？图表类型是否恰当？
□ 报告是否匹配受众（业务 vs 技术）？
□ 是否给出了可行动的建议（不只是数据陈述）？
□ SQL 是否可复现？是否注明口径和假设？
□ 经验是否沉淀到 field-journal？
```

## 常见坑（跨 skill 通用）

```text
1. 分析目标不清就动手，做完发现答非所问
2. 指标口径不对齐，不同人看同一数字理解不同
3. 数据质量未检查，结论建立在脏数据之上
4. 只给数据不给结论，决策者不知道该干嘛
5. 相关性当因果性，A/B 测试不严谨
6. 样本量不足就下结论，置信度不够
7. 忽略 Simpson 悖论，分组看和总体看结论相反
8. 图表误导（Y 轴截断 / 3D 饼图 / 双轴误导）
9. SQL 不可复现，换个人跑结果不一样
10. 报告只面向一种受众，CEO 和开发看同一份
11. 一次性分析不沉淀，下次重复造轮子
12. 忽略季节性和外部因素，把噪声当趋势
13. A/B 测试提前看结果，p-hacking
14. 指标太多没有优先级，什么都追等于什么都没追
```

具体 skill 内的常见坑见各 skill 的 SKILL.md。

## 多任务与中断处理

```text
1. 多主题并行：每个分析独立维护 SQL + 结论 + 报告，避免口径混乱
2. 中途中断：保存当前进度（已完成查询 + 待验证结论 + 待补数据）
3. 紧急分析：暂停常规工作，按优先级抢占处理（线上异常归因优先）
4. 数据阻塞（缺表 / 缺埋点 / 数据延迟）：记录阻塞原因 + 给出替代方案
```

---

## 数据伦理与合规

```text
1. PII 数据不出现在报告和仪表盘中
2. 分析结论不歧视特定用户群体
3. A/B 测试不对用户造成不可逆伤害
4. 数据访问权限最小化
5. 分析过程可审计、可追溯
6. 遵守 GDPR / CCPA 等适用法规
```

## 工具与自举

### 工具索引

工具状态以 `tool-index.md` 和 `tool-index.json` 为准。不要猜测版本号。

### 工具自举

缺少工具时调用（路径相对于项目根）：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File "scripts/bootstrap-project.ps1" -Workflow data-analyst
```

### 工具失败处理

| 情况 | 处理方式 |
|------|---------|
| bootstrap 成功 | 继续任务 |
| bootstrap 失败，原因明确 | 输出结构化引导（问题/原因/步骤/验证命令），等用户确认 |
| 同一工具失败 2 次 | 停止重试，给完整手动步骤 |
