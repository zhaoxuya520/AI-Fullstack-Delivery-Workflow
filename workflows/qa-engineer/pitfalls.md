# 测试工程师工作流常见坑

跨 skill 通用的高频坑，具体 skill 内的坑见各 SKILL.md。

## 1. 验收标准不清就开测

### 表现

PRD 没有明确验收标准，QA 自己揣摩"应该这样测"，开发觉得"我做的对"，PM 觉得"不是我要的"。

### 风险

- 测试用例覆盖错方向
- 大返工：50%+ 用例需要重写
- 三方都觉得对方在敷衍

### 避免方式

```text
1. 在 PRD 评审时同步定义验收标准
2. 验收标准用 Given-When-Then 格式
3. 每条 PRD 至少 1 条 P0 验收标准
4. PM / Dev / QA 三方签字
```

## 2. 只测成功路径

### 表现

测试用例 80% 是"用户输入正常数据 → 系统正常响应"，失败路径只有几条。

### 风险

- 50%+ Bug 在反向路径未发现
- 上线后用户用"不正常"操作触发 Bug
- 安全漏洞通常在错误处理层

### 避免方式

```text
失败路径必须覆盖：
  - 参数校验失败（必填、类型、长度、枚举）
  - 认证失败（无 Token / 过期 / 篡改）
  - 权限失败（无权限 / 跨用户 / 跨租户）
  - 资源不存在（404 / 410）
  - 资源冲突（409）
  - 业务校验失败（422）
  - 限流（429）
  - 服务异常（500）
```

---

## 3. 边界值漏测

### 表现

只测中间值，漏 0 / 1 / 最大 / 最大+1 / 负数 / 空字符串 / null。

### 风险

- 经典越界：数组下标越界、整数溢出
- 经典空值：null pointer
- 经典编码：UTF-8、Emoji、RTL

### 避免方式

```text
每个输入必测：
  - 边界外（min-1, max+1）
  - 边界（min, max）
  - 空值（null, "", "  ")
  - 特殊字符（'<>{}\\、Emoji、Unicode）
  - 超长字符串
  - 编码不一致（UTF-8 vs GBK）
```

---

## 4. 状态机非法转换不测

### 表现

只测合法转换（draft → submitted → paid），不测非法转换（已支付订单再次支付）。

### 风险

- 业务漏洞：重复扣款、状态错乱
- 数据完整性问题
- 资金损失

### 避免方式

```text
对每个状态机：
  1. 列出所有合法转换 → 全部测
  2. 列出所有非法转换 → 全部测
  3. 列出所有终态尝试再变更 → 全部测
```

---

## 5. 权限测试只测自己角色

### 表现

QA 用 user 账号测，没用 manager / admin / 跨租户测。

### 风险

- 越权漏洞（A 用户能改 B 用户数据）
- 跨租户数据泄露
- 字段级权限失效（普通用户看到内部字段）

### 避免方式

```text
对每个端点：
  - guest（未登录）
  - user（自己资源）
  - user（他人资源）
  - user（跨租户）
  - manager（团队资源）
  - admin（任意资源）
  - Token 过期 / 篡改
```

## 6. Bug 缺复现步骤

### 表现

Bug 报告："登录有问题，看图。" 开发：???

### 风险

- 开发反复回退："不能复现"
- 修复时间翻倍
- QA 和开发互相推卸

### 避免方式

```text
每个 Bug 必须包含：
  □ 标题：模块 + 现象 + 关键条件
  □ 环境（版本 / 浏览器 / 设备）
  □ 账号 / 角色
  □ 前置条件
  □ 复现步骤（最小化）
  □ 预期结果
  □ 实际结果
  □ 截图 / 日志 / trace_id
  □ 复现率（100% / 50% / 偶发）
  □ 严重度 + 优先级
```

## 7. "严重 Bug" 凭感觉打

### 表现

QA 拍 S0、PM 拍 S3，开发不知道按谁的处理。

### 风险

- 严重 Bug 延后修，造成更大影响
- 轻微 Bug 反复打扰开发
- 优先级混乱，进度受影响

### 避免方式

```text
严重度（QA 评估）：
  S0 致命：数据丢失 / 资金损失 / 全用户阻塞
  S1 严重：核心功能不可用 / 部分用户阻塞
  S2 一般：功能不完整 / 有 workaround
  S3 轻微：UI / 文案 / 边缘场景

优先级（PM 评估）：
  P0 阻塞上线
  P1 本版本必修
  P2 下版本修
  P3 何时都行

注意：S 和 P 不必相同
  S3 文案错出现在落地页 → P0
```

## 8. 回归套件无差别全跑

### 表现

每次发布跑 3 小时回归，CI 卡死，开发怨声载道。

### 风险

- CI 时间爆炸
- 频繁失败 / Flaky 用例淹没真问题
- 团队不愿跑 → 干脆跳过

### 避免方式

```text
三层套件：
  Smoke（5 min）：CI 必跑
  Sanity（30 min）：发布前
  Full（1~3 h）：重大版本

按变更影响选套件：
  Bug 修复 → Smoke + 该 Bug 用例
  单模块改 → Smoke + Sanity（该模块）
  跨模块 → Smoke + Sanity（涉及模块）
  重构 → Full
  依赖升级 → Full
```

## 9. 测试数据手工造

### 表现

用例需要"已支付订单"，QA 手动在数据库 INSERT。

### 风险

- 下次环境重置全没了
- 不可复现
- 数据格式不一致
- 团队成员各造各的

### 避免方式

```text
1. 工厂模式（Factory）程序化生成
2. Fixture 文件进版本控制
3. 测试账号文档化
4. 自动 setup / teardown
5. 数据规模可参数化
```

---

## 10. 性能问题留给生产发现

### 表现

测试环境不做性能测试，上线后才发现 P99 5 秒。

### 风险

- 生产事故
- 紧急回滚
- 用户流失

### 避免方式

```text
上线前必做：
  □ 性能基线（P50 / P95 / P99）
  □ 与上版本对比
  □ 浸泡测试（≥ 1 小时）
  □ 尖峰测试
  □ 容量结论给 SRE

测试数据规模：接近生产 80%~100%
```

---

## 11. 第三方依赖没策略

### 表现

测试时第三方 API 时通时不通，CI 错误率 30%。

### 风险

- CI 不可靠 → 团队失去信心
- 真问题被淹没
- 阻塞合并

### 避免方式

```text
1. Mock：稳定但不真实
2. 录制回放（VCR）：真实但需更新
3. 沙箱环境：真实第三方提供的测试环境
4. Contract Testing：与第三方约定契约
5. 降级策略：第三方挂时单独用例
```

---

## 12. 同 Bug 反复出现

### 表现

修了又出，出了又修。

### 风险

- 浪费修复时间
- 用户体验恶化
- 团队失去信任

### 避免方式

```text
每个 Bug 修复后必做：
  1. 写最小复现回归用例
  2. 写边界变体用例
  3. 加入 Sanity 套件
  4. 标注：源自 Bug-XXX
  5. 6 个月内不删
  6. 探索同类风险（"还有哪里可能有这问题？"）
```

---

## 13. 测试报告只列数字不给结论

### 表现

报告："145/145 通过率 98%"。决策者：???

### 风险

- 决策者不知道能不能放行
- 不会再看测试报告
- QA 价值被低估

### 避免方式

```text
首页必须有：
  ✅ 建议放行 / ⚠️ 有条件放行 / ❌ 不建议放行

附：
  - 关键风险（1-2 条）
  - 已知问题（业务方已知悉）
  - 放行 Checklist 状态
```

---

## 14. 质量门禁靠口头约定

### 表现

"我们规定 S0 不能上线"——但没自动化检查，靠记忆。

### 风险

- 紧急时被"特殊处理"
- 团队各有各的标准
- 漏放 Bug 后无法追溯

### 避免方式

```text
1. 门禁规则写成 YAML / 文档
2. CI/CD 自动化检查
3. 失败有清晰错误信息
4. 紧急通道明确（谁能批准 / 后续要求）
5. 月度复盘门禁有效性
```
