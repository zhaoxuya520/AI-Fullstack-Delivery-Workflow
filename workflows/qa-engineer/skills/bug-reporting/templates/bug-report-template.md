# Bug 报告模板

## 1. Bug 标题

格式：`[模块] [现象] - [关键条件]`

示例：
- `[订单] 退款金额可输入负数 - 所有支付方式`
- `[支付] 重复点击提交导致重复扣款 - 网络慢时`

---

## 2. 基本信息

```text
Bug ID：BUG-XXX
报告人：
报告日期：
环境：dev / staging / pre-prod / production
版本：v1.2.3 / commit abc1234
浏览器/设备：Chrome 124 / iOS 18 / Android 14
账号 / 角色：user_a / role=user, tenant=A
```

---

## 3. 复现步骤（最小复现）

```text
前置：
  1. 已登录账号 user_a
  2. 已存在订单 ORD-123（状态=paid）

步骤：
  1. 进入订单详情页 /orders/ORD-123
  2. 点击 "退款"
  3. 在退款金额输入 -100
  4. 点击 "确认退款"

预期：
  返回错误码 INVALID_AMOUNT，订单状态不变

实际：
  返回 200 OK，订单余额变为 -100，触发了退款回调
```

## 5. 严重度 × 优先级

| 严重度（影响） | 等级 |
|---|---|
| 数据丢失 / 资金损失 / 全用户阻塞 | S0（致命） |
| 核心功能不可用 / 部分用户阻塞 | S1（严重） |
| 功能不完整 / 有 workaround | S2（一般） |
| UI 错误 / 文案错误 / 边缘场景 | S3（轻微） |

| 优先级（紧急度） | 等级 |
|---|---|
| 立即修，阻塞上线 | P0 |
| 本版本修 | P1 |
| 下版本修 | P2 |
| 排期看 | P3 |

本 Bug：S1 × P0

## 6. 影响范围

```text
受影响用户：所有支付用户 / 仅特定角色 / 仅边缘场景
受影响功能：退款 / 订单 / 计费
受影响环境：production / staging
是否需要立即缓解：是 / 否
缓解措施：禁用前端入口 / 后端拦截 / 限流
```

## 11. 自检（提交前）

```text
□ 标题包含模块 + 现象 + 条件
□ 复现步骤是最小路径
□ 有预期 vs 实际对比
□ 严重度和优先级有依据
□ 截图 / 日志 / trace_id 完整
□ 复现率明确
□ 影响范围清楚
□ 是否需要立即缓解已说明
```
