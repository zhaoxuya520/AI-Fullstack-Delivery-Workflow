# 测试方法论深度指南

本文档是 QA 工程师的核心方法论参考，涵盖测试用例设计、测试计划、Bug 报告、测试策略等关键实践。

---

## 1. 测试用例设计方法

### 1.1 等价类划分（Equivalence Partitioning）

将输入域划分为若干等价类，每个类中取一个代表值测试。

**原则：**
- 有效等价类：符合规格的输入
- 无效等价类：不符合规格的输入
- 每个等价类至少一个测试用例

**示例：年龄输入（1~150）**

| 等价类 | 范围 | 代表值 | 预期 |
|--------|------|--------|------|
| 有效 | 1~150 | 25 | 通过 |
| 无效-低于下界 | <1 | 0, -1 | 报错 |
| 无效-高于上界 | >150 | 151, 999 | 报错 |
| 无效-非数字 | 字母/特殊字符 | "abc", "@#" | 报错 |
| 无效-空 | null/空字符串 | "", null | 报错 |

**用例设计模板：**

```markdown
TC-EP-001: 年龄输入 - 有效等价类
  前置条件: 用户在注册页面
  输入: age = 25
  操作: 提交表单
  预期: 注册成功
  等价类: 有效-正常范围
```

### 1.2 边界值分析（Boundary Value Analysis）

在等价类边界取值，包括边界上、边界内外各一个值。

**经典边界值：**
- min - 1（无效）
- min（有效边界）
- min + 1（有效）
- 正常值（有效）
- max - 1（有效）
- max（有效边界）
- max + 1（无效）

**示例：密码长度（8~20 字符）**

| 测试点 | 长度 | 预期结果 |
|--------|------|---------|
| min-1 | 7 | 报错：密码太短 |
| min | 8 | 通过 |
| min+1 | 9 | 通过 |
| normal | 12 | 通过 |
| max-1 | 19 | 通过 |
| max | 20 | 通过 |
| max+1 | 21 | 报错：密码太长 |
| 空 | 0 | 报错：密码不能为空 |

### 1.3 判定表法（Decision Table）

适用于多条件组合的业务规则。

**示例：订单折扣规则**

| 条件 | R1 | R2 | R3 | R4 | R5 |
|------|----|----|----|----|----| 
| VIP 会员 | Y | Y | N | N | N |
| 订单 > 500 | Y | N | Y | N | N |
| 有优惠券 | - | - | - | Y | N |
| **动作** | | | | | |
| 打 8 折 | ✓ | | | | |
| 打 9 折 | | ✓ | ✓ | | |
| 减 50 | | | | ✓ | |
| 无折扣 | | | | | ✓ |

**转化为测试用例：**

```markdown
TC-DT-001: VIP + 订单>500 → 8 折
  前置: 用户为 VIP，购物车金额 600 元
  操作: 提交订单
  预期: 实付 480 元（600 × 0.8）

TC-DT-002: VIP + 订单≤500 → 9 折
  前置: 用户为 VIP，购物车金额 300 元
  操作: 提交订单
  预期: 实付 270 元（300 × 0.9）
```

### 1.4 状态转换法（State Transition）

适用于有明确状态流转的业务对象。

**示例：订单状态机**

```text
[待支付] --支付成功--> [已支付] --发货--> [已发货] --确认收货--> [已完成]
    |                      |                    |
    |--超时未付-->  [已取消]  |--申请退款-->  [退款中] --退款成功--> [已退款]
    |                      |
    |--用户取消-->  [已取消]  |--商家取消-->  [已取消]
```

**测试用例矩阵：**

| 当前状态 | 事件 | 目标状态 | 是否合法 |
|---------|------|---------|---------|
| 待支付 | 支付成功 | 已支付 | ✓ 合法 |
| 待支付 | 用户取消 | 已取消 | ✓ 合法 |
| 待支付 | 确认收货 | - | ✗ 非法，应报错 |
| 已支付 | 再次支付 | - | ✗ 非法，幂等或报错 |
| 已取消 | 支付成功 | - | ✗ 非法，应报错 |
| 已发货 | 确认收货 | 已完成 | ✓ 合法 |
| 已完成 | 申请退款 | - | ✗ 超时不可退 |

### 1.5 错误猜测法（Error Guessing）

基于经验猜测容易出错的场景。

**高频错误场景清单：**

```text
输入类：
  - 空字符串 / null / undefined
  - 超长字符串（超过数据库字段长度）
  - 特殊字符（<script>、SQL 注入、emoji）
  - 前后空格
  - 零值 / 负数
  - 浮点精度（0.1 + 0.2）

并发类：
  - 同时提交同一表单
  - 重复点击按钮
  - 同一资源多人同时编辑
  - 库存扣减竞争

网络类：
  - 请求超时
  - 断网恢复后重新提交
  - 弱网环境（3G / 高延迟）

权限类：
  - 越权访问他人数据
  - Token 过期后操作
  - 角色降级后访问高权限功能

数据类：
  - 大数据量分页（第 10000 页）
  - 空列表
  - 单条数据
  - 刚好等于分页大小的数据量
```

## 3. Bug 报告标准格式

```markdown
# Bug 报告

## 基本信息
- Bug ID: BUG-2024-001
- 标题: [模块] 简洁描述现象（不超过 50 字）
- 严重度: P0-阻塞 / P1-严重 / P2-一般 / P3-轻微
- 优先级: 紧急 / 高 / 中 / 低
- 模块: 订单 / 用户 / 支付 / ...
- 环境: staging / prod / dev
- 版本: v2.3.1
- 报告人: xxx
- 指派给: xxx

## 复现步骤
1. 登录系统（账号：test@example.com）
2. 进入「订单列表」页面
3. 点击「筛选」→ 选择日期范围 2024-01-01 ~ 2024-01-31
4. 点击「导出」按钮

## 预期结果
- 下载包含筛选范围内订单的 Excel 文件

## 实际结果
- 页面报错 500，控制台显示 "TypeError: Cannot read property 'map' of undefined"
- 截图：[附件]

## 附加信息
- 浏览器: Chrome 120
- 分辨率: 1920×1080
- 网络: 正常
- 数据量: 筛选结果约 5000 条
- 备注: 数据量 < 100 条时可正常导出

## 根因分析（可选）
- 疑似大数据量导致后端超时，未做分页导出
```

**严重度定义矩阵：**

| 严重度 | 定义 | 示例 |
|--------|------|------|
| P0-阻塞 | 核心功能不可用，无替代方案 | 无法登录、支付失败、数据丢失 |
| P1-严重 | 核心功能异常，有临时替代方案 | 列表不显示但 API 正常、导出失败但可手动复制 |
| P2-一般 | 非核心功能异常或体验差 | 排序不生效、图片加载慢、样式错位 |
| P3-轻微 | 不影响使用的小问题 | 文案错别字、多余空行、颜色偏差 |

## 4. 测试优先级决策（风险驱动测试）

### 风险矩阵

```text
影响度
  高  | 中优先  | 高优先  | 最高优先 |
  中  | 低优先  | 中优先  | 高优先   |
  低  | 最低    | 低优先  | 中优先   |
       --------+---------+----------
         低       中        高
                 发生概率
```

### 优先级排序框架

**STEP 1: 识别风险因素**

| 维度 | 高风险信号 |
|------|-----------|
| 复杂度 | 多条件组合、状态流转复杂、涉及金额计算 |
| 变更频率 | 新功能、重构代码、第三方升级 |
| 历史缺陷 | 同模块历史 Bug 多、回归 Bug |
| 影响范围 | 核心业务路径、高频操作、大量用户 |
| 数据敏感 | 涉及支付、个人信息、权限 |

**STEP 2: 按风险分配测试投入**

```text
高风险区（50% 测试时间）：
  - 支付流程
  - 权限控制
  - 数据一致性
  - 核心业务逻辑

中风险区（30% 测试时间）：
  - 列表/搜索/筛选
  - 表单提交
  - 文件上传/导出

低风险区（20% 测试时间）：
  - 静态展示页面
  - 帮助文档
  - 不常用功能
```

## 5. 回归测试策略

### 回归范围确定

```text
1. 变更影响分析：
   - 直接变更的代码 → 必须回归
   - 调用方 / 被调用方 → 评估回归
   - 共享数据的模块 → 评估回归
   - 无关模块 → 不回归

2. 回归套件分层：
   - 冒烟回归（5~10 条）：核心路径通过即可部署
   - 基础回归（30~50 条）：每日构建后执行
   - 完整回归（全部）：版本发布前执行
```

### 回归用例选择策略

| 策略 | 适用场景 | 方法 |
|------|---------|------|
| 基于变更 | 功能修改/Bug修复 | 只回归受影响的功能 |
| 基于风险 | 核心模块变更 | 优先回归高风险用例 |
| 基于时间 | 时间紧张 | 只跑冒烟 + 高优先级 |
| 全量回归 | 大版本发布 | 跑完整回归套件 |

### 回归套件维护

```text
每个 Sprint 结束时：
  1. 新增用例：本 Sprint 的核心场景
  2. 删除用例：已废弃功能的用例
  3. 更新用例：变更的业务规则
  4. 标记优先级：基于最近 Bug 分布调整

回归套件膨胀警告：
  - 套件执行时间 > 4 小时 → 必须剪枝
  - 同一功能 > 20 条回归用例 → 合并/精简
  - 连续 10 次全部通过的用例 → 降级到完整回归
```

## 6. 验收测试清单设计

### Given-When-Then 格式

```gherkin
Feature: 用户登录

  Scenario: 正常登录
    Given 用户已注册，邮箱 "user@example.com"，密码 "Pass1234"
    When 用户在登录页输入正确的邮箱和密码并点击登录
    Then 跳转到首页
    And 显示用户头像和昵称
    And 后端返回有效 Token（JWT，有效期 7 天）

  Scenario: 密码错误
    Given 用户已注册，邮箱 "user@example.com"
    When 用户输入错误密码 "WrongPass" 并点击登录
    Then 显示错误提示 "邮箱或密码错误"
    And 不跳转页面
    And 连续 5 次错误后锁定 15 分钟

  Scenario: 账号不存在
    Given 邮箱 "noone@example.com" 未注册
    When 用户输入该邮箱并点击登录
    Then 显示错误提示 "邮箱或密码错误"（不泄露账号是否存在）
```

### 验收清单模板

```markdown
## 功能验收清单：[功能名称]

### 核心路径
- [ ] 主流程完整走通（Happy Path）
- [ ] 主流程各步骤有正确反馈

### 异常路径
- [ ] 必填字段为空
- [ ] 输入超出范围
- [ ] 网络中断/超时
- [ ] 并发操作
- [ ] 权限不足

### 状态完整性
- [ ] 空状态（无数据时的展示）
- [ ] 加载状态
- [ ] 错误状态
- [ ] 成功状态
- [ ] 部分成功状态

### 兼容性
- [ ] 目标浏览器/设备
- [ ] 响应式断点
- [ ] 键盘操作
```

## 7. 探索性测试 Session 方法

### SBTM（Session-Based Test Management）

```markdown
## 探索性测试 Session 记录

### Session 信息
- Charter: 探索[模块名]的[场景]，寻找[类型]缺陷
- 测试者: xxx
- 时间: 60 分钟（计划）/ 55 分钟（实际）
- 环境: staging

### 测试笔记（时间线）
- 00:00 - 开始探索订单列表页面
- 05:00 - 尝试大量筛选条件组合
- 12:00 - 发现：日期范围选今天之后的日期，列表显示空但无提示
- 18:00 - 尝试极端数据量，10000 条记录时翻到最后一页
- 25:00 - 发现：最后一页显示 0 条但页码仍可点击
- 35:00 - 测试导出功能，大数据量导出
- 42:00 - 发现：导出 > 5000 条时超时 500 错误
- 50:00 - 验证边界：4999 条可以导出，5000 条失败

### Bug 发现
1. BUG-001: 日期筛选未来日期无提示
2. BUG-002: 分页最后一页异常
3. BUG-003: 大数据量导出超时

### Session 指标
- Bug 数: 3
- 测试覆盖: 订单列表筛选 + 分页 + 导出
- 剩余风险: 多条件组合筛选未充分覆盖
```

### 探索性测试启发式

**CRUD 启发式：**
```text
C - Create: 创建重复数据、创建后立即编辑、批量创建
R - Read: 空列表、大列表、筛选后为空、排序边界
U - Update: 并发编辑、部分更新、更新为空值
D - Delete: 删除被引用的数据、批量删除、删除后撤销
```

**FEW HICCUPS 启发式：**
```text
F - Flow: 用户流程是否顺畅
E - Entity: 数据实体是否一致
W - Words: 文案是否准确
H - Help: 帮助信息是否有用
I - Input: 输入验证是否完整
C - Config: 配置是否生效
C - Compatibility: 兼容性
U - Usability: 可用性
P - Performance: 性能感受
S - Security: 安全基础检查
```

## 8. 移动端/小程序测试要点

### 移动端专项测试清单

```text
基础功能：
  - [ ] 触摸操作（点击、滑动、长按、双击、捏合）
  - [ ] 横竖屏切换
  - [ ] 键盘弹出/收起时的页面布局
  - [ ] 系统返回键行为
  - [ ] 手势冲突（滑动和列表滚动）

中断与恢复：
  - [ ] 来电中断后恢复
  - [ ] 切换到后台再返回
  - [ ] 锁屏再解锁
  - [ ] 低电量模式
  - [ ] 推送通知弹出时的交互

网络：
  - [ ] WiFi ↔ 4G/5G 切换
  - [ ] 弱网（2G/3G 模拟）
  - [ ] 断网后恢复
  - [ ] 请求超时处理

设备差异：
  - [ ] 不同屏幕尺寸（小屏 iPhone SE / 大屏 iPhone Pro Max）
  - [ ] 刘海屏/折叠屏/平板适配
  - [ ] 不同 OS 版本（iOS 15+ / Android 10+）
  - [ ] 深色模式

小程序专项：
  - [ ] 冷启动/热启动时间
  - [ ] 页面栈深度（超过 10 层）
  - [ ] 分享卡片内容和跳转
  - [ ] 授权弹窗（位置/相册/通讯录）拒绝后处理
  - [ ] 小程序更新提示
  - [ ] 胶囊按钮区域不被遮挡
```

### 性能关注点

```text
启动时间：冷启动 < 3s，热启动 < 1s
页面加载：首屏 < 2s，列表滚动无卡顿
内存占用：不超过系统警告线
电池消耗：后台不持续 GPS/网络
包体积：小程序主包 < 2MB，分包 < 20MB
```

## 9. 常见测试场景速查表

### 登录场景

| 场景 | 测试点 | 预期 |
|------|--------|------|
| 正常登录 | 正确邮箱+密码 | 登录成功，跳转首页 |
| 密码错误 | 错误密码 | 提示错误，不泄露账号存在 |
| 账号锁定 | 连续 5 次错误 | 锁定 15 分钟 |
| Token 过期 | 等待 Token 失效 | 跳转登录页 |
| 多设备登录 | 不同设备登录 | 按策略踢出/共存 |
| SQL 注入 | 输入 `' OR 1=1 --` | 不泄露数据 |
| XSS | 输入 `<script>alert(1)</script>` | 不执行脚本 |
| 记住密码 | 勾选后关闭重开 | 自动填充 |
| 第三方登录 | 微信/Google OAuth | 正常绑定和登录 |

### 支付场景

| 场景 | 测试点 | 预期 |
|------|--------|------|
| 正常支付 | 足够余额 | 扣款成功，订单状态变更 |
| 余额不足 | 余额 < 订单金额 | 提示余额不足 |
| 重复支付 | 快速点击两次 | 幂等，只扣一次 |
| 支付超时 | 等待超时 | 自动取消，退回库存 |

| 优惠券 + 积分 | 多种优惠叠加 | 计算正确，不超扣 |
| 跨境支付 | 不同币种 | 汇率正确，金额无精度丢失 |
