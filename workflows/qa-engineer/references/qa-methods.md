# QA 综合方法

## 核心目标

QA 工作流不是只找 Bug，而是用可验证证据回答：当前版本是否达到可交付质量。

## 标准 QA 链路

```text
理解需求和验收标准
→ 定义测试范围
→ 按风险排序
→ 设计用例
→ 准备测试数据
→ 执行测试
→ 记录缺陷
→ 回归验证
→ 输出测试报告
→ 质量门禁放行判断
```

## 方法组合

| 目标 | 方法 |
|------|------|
| 明确测什么 | 测试策略、需求追踪矩阵、验收标准映射 |
| 明确先测什么 | 风险矩阵、变更影响分析、核心路径优先 |
| 提高用例质量 | 等价类、边界值、状态转换、权限矩阵、错误路径 |
| 提高缺陷可修复性 | 最小复现、环境信息、期望/实际、证据、严重度/优先级矩阵 |
| 控制发布风险 | 回归套件、冒烟测试、UAT、质量门禁、已知问题签字 |
| 改进长期质量 | 缺陷根因分析、趋势分析、field-journal 经验沉淀 |

## 质量信号

- 必测用例通过率
- S0/S1 缺陷数
- 回归套件通过率
- 需求/验收标准覆盖率
- API 契约和错误路径覆盖
- 性能 SLO 达标情况
- 已知风险是否有业务确认
- 回滚、监控、灰度是否就绪

## 放行原则

- 没有明确结论的测试报告不可作为放行依据。
- S0/S1 未关闭时默认不放行，除非有明确业务签字和降级方案。
- 测试覆盖不足必须写入风险说明，不能用“未发现问题”替代。
- 质量门禁必须明文化，不能依赖口头约定。
