# 工作流协作图

## 1. 主链路

```text
产品经理
   ↓
项目经理
   ↓
UI/UX 设计
   ↓
API 设计 ─────→ 前端工程师
   ↓                ↑
数据库工程师 ─→ 后端工程师
   ↓                ↓
测试工程师 ←── 全栈工程师
   ↓
自动化测试
   ↓
安全工程师 ──→ 逆向/渗透
   ↓
DevOps 工程师
   ↓
SRE/运维
   ↓
数据分析 / AI算法
   ↓
技术文档
   ↓
field-journal
```

## 2. 常见协作模式

### 新功能开发

```text
产品经理 → API设计 + 数据库 → 后端 → 前端 → QA → 文档
```

### MVP 快速交付

```text
产品经理 → 全栈工程师 → QA → DevOps → 文档
```

### 上线前检查

```text
QA → 安全工程师 → DevOps → SRE → 技术文档
```

### 安全专项

```text
安全工程师
  ↓
逆向/渗透工作流
  ↓
后端/前端/DevOps 修复
  ↓
QA 回归
  ↓
技术文档
```

### 线上故障

```text
SRE/运维 → 后端/前端/数据库 → QA回归 → DevOps发布 → 复盘文档
```

### 数据智能功能

```text
产品经理 → 数据分析 → AI/算法 → 后端 → 前端 → QA → SRE
```

## 3. 单工作流触发条件

任务满足以下条件时，可只进入一个工作流：

- 输入清楚
- 影响范围明确
- 产物单一
- 不涉及跨端联调
- 不影响部署或安全边界

## 4. 多工作流触发条件

满足以下任一条件时，应使用多工作流协同：

- 涉及前后端联调
- 涉及数据库结构变化
- 涉及权限、安全或敏感数据
- 涉及生产部署
- 涉及线上故障
- 需要正式报告或复盘
